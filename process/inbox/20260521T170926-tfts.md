# TFTS — 2026-05-21T17:09:26Z

problem: Station template line.json files had null lengths and no geometry (startPoint, endPoint, segments). Show Routes could not draw station tracks. The populate tool wrote nothing to disk through multiple sessions.

fault_ref: 20260521T170926

arc:
  - try: Extract centerlines using bbox on world-space placed instances via collect_station_tracks
    result: failed — splattering; wrong dominant axis when track separation > stub length; bbox in world space mixed station and inter-station geometry
    revealed: bbox axis selection fails when the perpendicular track spacing is larger than the track length (e.g. 3.5m separation vs 2m stub)

  - try: Search model.definitions by formation_id name to find the station component definition
    result: failed — all station definitions are anonymous (Group#N); formation_id is stored as a JPods attribute on the placed instance, not as the definition name
    revealed: SketchUp stores formation templates as Group definitions with auto-generated names; you cannot look them up by formation_id

  - try: Search model.entities for name == 'JPods Structure' to find placed stations
    result: failed — no output; stations are nested inside a SU_InstanceSet geolocation wrapper at depth 1, invisible to a depth-0 search
    revealed: Geolocated models have all user content one level deep inside a SU_InstanceSet component; model.entities.each only reaches the wrapper

  - try: Open each template model.skp manually and run populate_from_open_template
    result: failed — _entity_tag undefined; the helper existed only in StructureTool, not in PathJSON; each module needs its own copy of shared helpers
    revealed: Ruby module methods are not shared between modules; a method defined in JPods::StructureTool is not callable as JPods::PathJSON._entity_tag

  - try: Use Sketchup.active_model.definitions.load(skp_path) to load each template model.skp as a component definition, index gw_* entities by layer/tag name, extract bbox centerlines in definition-local space
    result: succeeded — 5/5 templates updated; station_line_end 12/12, station_thru_dip 12/12, JPods_station_parking 4/6 (2 tag mismatches), station_solar 6, traffic_circle7 6
    revealed: definitions.load() is the correct entry point; the loaded definition's entity tree is in local template space (no world transform confusion); layer name IS the gw_* tag; both instance name and layer name are set to the same gw_* value in all templates

principle: To extract geometry from a station template, load the model.skp via definitions.load() and index entities by SketchUp layer (tag) name. Do not search model.definitions by formation name (anonymous), do not search model.entities (depth-0 misses geolocation wrapper), do not rely on methods from other modules. Each JPods module that needs entity tag lookup must define its own _entity_tag reading e.layer.name.

domain: SU

why_it_was_hard:
  1. Three different location assumptions were wrong in sequence: definition name, entity name, depth.
  2. The SketchUp geolocation wrapper (SU_InstanceSet) is invisible unless you know to look past depth 0.
  3. Station templates are Group definitions (group=true), not named ComponentDefinitions — the distinction only becomes visible by dumping all definitions and seeing Group#N everywhere.
  4. _entity_tag existed in StructureTool from an earlier session; the assumption that it was available in PathJSON was never validated.
  5. The constant redefinition warnings on reload obscured the real error in early runs.
  6. The fix required understanding three separate SketchUp concepts simultaneously: definition loading, group vs component, and layer/tag naming.
