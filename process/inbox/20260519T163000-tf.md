# TF — 2026-05-19T16:30:00
summary: SketchUp entity references become invalid ("reference to deleted Group") after adding geometry to a sibling group during the same operation — always guard with entity.deleted? before re-accessing a held reference
code:    jpod_animator.rb show_trip_path_by_id (gap handling loop ~line 3647)

## What became clear

When iterating a gap list and placing gap circles in a new `gaps_grp`, the group
object held in `last_grp` (a 'JPods Guideway' group set earlier in the same loop)
became invalid by the time the second gap iteration called `last_grp.bounds.center`.

The exact mechanism is unclear — SketchUp's geometry kernel may sweep or reindex
entity references when `add_face` resolves coplanar/coincident geometry inside a
sibling group. Whatever the cause, the symptom is consistent: a reference that was
valid two lines ago throws "reference to deleted Group".

## The rule

**Never re-access a held SketchUp entity reference without an `entity.deleted?` guard
if any model mutation (add_group, add_face, add_circle, erase!) has occurred since
the reference was set.**

```ruby
# WRONG — crashes if SketchUp invalidated last_grp after add_face:
pt = last_grp.bounds.center

# RIGHT:
if last_grp && !last_grp.deleted?
  pt = last_grp.bounds.center
  # ... use pt ...
else
  last_grp = nil   # clear dead reference — don't carry it forward
end
```

## Also applies to aggregated collections

Any array of entity references collected before model mutations must be filtered
before use:

```ruby
colored_groups.reject!(&:deleted?)
add_labels_for_mode(model, colored_groups, 'trip')
```

## Domain

[SketchUp] — applies anywhere entity references are held across model mutations
within a single `start_operation` block. Especially risky in loops that both
collect references AND add geometry.
