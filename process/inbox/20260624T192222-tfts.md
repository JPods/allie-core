## Cross-Domain Lessons for Agent Fleet

This TFTS applies directly to every future system where an agent (Nora, Natalie, Noelle, Sally, Alice) runs its own processor and needs to register MCP tools with a host. The pattern generalizes:

### Lesson 1 — Config File Jurisdiction

A host application may split configuration across multiple files with no overlap warning. The file that "looks right" (settings.json — it holds every other config category) may not govern the subsystem you're configuring. **Never assume config file jurisdiction from naming or proximity. Verify by reading the host's own registration command.**

Applies to: Any agent registering tools with a host (Claude Code, WebClerk, a future JPods dispatcher). The registration mechanism defines where config lives, not the config file you already know about.

### Lesson 2 — Silent Misconfiguration Is the Worst Failure Mode

The mcpServers block in settings.json produced no error, no warning, no log entry. The tools simply didn't exist. This is worse than a crash — a crash tells you something is wrong.

**Design rule for JPods agents:** When an agent registers a capability with a host and the host doesn't acknowledge it, the agent must detect and report the gap. Don't let "tool registered but never called" be silent. Nora registering a sensor with Natalie, Sally registering a slot with Noelle — if the host never calls back during the handshake window, that's a FAULT, not normal.

### Lesson 3 — Verify at the Host, Not at the Agent

Testing allie-mcp-server.py directly proved the server worked. But the server wasn't the problem — the host (Claude Code) never spawned it. **Verification must happen from the host's perspective** (`claude mcp list`), not the agent's.

Applies to: Pod startup. Testing that a pod's MQTT handler works locally doesn't prove Natalie can reach it. Test from Natalie's side: can she dispatch a trip to this pod? If not, the registration path is broken regardless of what the pod thinks.

### Lesson 4 — Session Lifecycle Governs Tool Availability

MCP servers load at Claude Code session startup. Adding a server mid-session doesn't make its tools available — you must restart. This is a session lifecycle constraint, not a bug.

Applies to: Hot-reload expectations across the fleet. If Noelle validates a network at startup, adding a new station mid-session may not be visible until the next validation cycle. Document which capabilities are session-scoped vs. live-updated. Never assume hot-reload unless the host explicitly supports it.

### Lesson 5 — Registration Command Is the Source of Truth

`claude mcp add` is the authoritative registration path. Not hand-editing JSON. Not copying config from another file. The command knows where to write, what format to use, and how to validate.

Applies to: All agent registration. If Nora registers with Natalie, there should be ONE registration path (an API call, a config command, a handshake protocol) — not "edit this JSON file and hope the host reads it." The registration mechanism is the contract.
