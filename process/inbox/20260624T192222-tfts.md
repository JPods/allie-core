## Cross-Domain Lessons for Agent Fleet

This TFTS applies directly to every future system where an agent (Nora, Natalie, Noelle, Sally, Alice) runs its own processor and needs to register tools with a host. The pattern generalizes.

Reviewed by Allie 2026-06-24. Agreements noted. One conflict resolved (Lesson 6).

### Lesson 1 — Config File Jurisdiction

A host application may split configuration across multiple files with no overlap warning. The file that "looks right" (settings.json — it holds every other config category) may not govern the subsystem you're configuring. **Never assume config file jurisdiction from naming or proximity. Verify by reading the host's own registration command.**

Applies to: Any agent registering tools with a host (Claude Code, WebClerk, a future JPods dispatcher). The registration mechanism defines where config lives, not the config file you already know about.

**Allie agrees.** Adds: every agent's docs must state the correct config file explicitly. Don't let a developer infer it from the file that "looks right."

### Lesson 2 — Silent Misconfiguration Is the Worst Failure Mode

The mcpServers block in settings.json produced no error, no warning, no log entry. The tools simply didn't exist. This is worse than a crash — a crash tells you something is wrong.

**Design rule for JPods agents:** When an agent registers a capability with a host and the host doesn't acknowledge it, the agent must detect and report the gap. Don't let "tool registered but never called" be silent. Nora registering a sensor with Natalie, Sally registering a slot with Noelle — if the host never calls back during the handshake window, that's a FAULT, not normal.

**Allie agrees.** Adds: the startup script should include a sanity check — e.g., `claude mcp list | grep -q Connected` — and exit with a clear message if nothing is connected. The check belongs in the startup path, not as an afterthought.

### Lesson 3 — Verify at the Host, Not at the Agent

Testing allie-mcp-server.py directly proved the server worked. But the server wasn't the problem — the host (Claude Code) never spawned it. **Verification must happen from the host's perspective** (`claude mcp list`), not the agent's.

Applies to: Pod startup. Testing that a pod's MQTT handler works locally doesn't prove Natalie can reach it. Test from Natalie's side: can she dispatch a trip to this pod? If not, the registration path is broken regardless of what the pod thinks.

**Allie agrees.** Adds: for agents on remote hosts (e.g., Alice's cloud VM), a local test run is not enough. Run the verification command on the host machine itself.

### Lesson 4 — Session Lifecycle Governs Tool Availability

MCP servers load at Claude Code session startup. Adding a server mid-session doesn't make its tools available — you must restart. This is a session lifecycle constraint, not a bug.

Applies to: Hot-reload expectations across the fleet. If Noelle validates a network at startup, adding a new station mid-session may not be visible until the next validation cycle. **Document which capabilities are session-scoped vs. live-updated for every agent.** Never assume hot-reload unless the host explicitly supports it.

**Allie agrees.** Strengthens: each agent's onboarding docs should have a table listing capabilities as either "session-scoped" (requires restart) or "live-updated" (takes effect immediately). If undocumented, assume session-scoped.

### Lesson 5 — Registration Command Is the Source of Truth

`claude mcp add` is the authoritative registration path. Not hand-editing JSON. Not copying config from another file. The command knows where to write, what format to use, and how to validate.

Applies to: All agent registration. If Nora registers with Natalie, there should be ONE registration path (an API call, a config command, a handshake protocol) — not "edit this JSON file and hope the host reads it." The registration mechanism is the contract.

**Allie agrees and sharpens:** Even if you hand-edit the JSON file correctly, the host may cache or ignore hand-edits. The registration command refreshes internal state that file edits alone may not trigger. Always use the command, even if you know the file format.

### Lesson 6 — Post-Registration Round-Trip Verification

*New — added from Allie's review.*

Checking `claude mcp list` verifies the host sees the server. But "Connected" does not prove the tools work end-to-end. After registration, make a test tool call (e.g., `ask_allie "test"`) to confirm the full round-trip: host → MCP server → backend (Ollama) → response.

Applies to: Every agent registration. Natalie confirming she can see Nora's MQTT topic is not enough — dispatch a test trip and confirm the response arrives. The handshake must include a round-trip proof, not just a connection check.

**Conflict resolved:** Allie suggested checking `~/.claude/logs/mcp.log` for connection errors. This file does not exist — Claude Code produces no MCP-specific log file (verified 2026-06-24). The absence of a log is itself an instance of Lesson 2 (silent failure). Until Claude Code adds MCP logging, the only diagnostic path is: `claude mcp list` (connection check) + test tool call (round-trip check). If both pass and tools still don't appear, restart the session (Lesson 4).

## Agent Fleet Startup Checklist

Consolidated from Claude Code's lessons and Allie's review. Applies to any agent registering MCP tools with a host.

1. **Register via the host's command** — not by hand-editing config files.
   `claude mcp add -s user <name> -- <command> [args]`

2. **Verify at the host** — run `claude mcp list` (or equivalent) and confirm "Connected."

3. **Test the round-trip** — make one real tool call and confirm a response arrives.

4. **Add a startup gate** — the agent's startup script must check registration status and fail loudly if disconnected. Silent absence is not acceptable.

5. **Document session scope** — state explicitly whether each capability is session-scoped (requires restart) or live-updated.

6. **Keep the MCP server alive** — if the backend (Ollama, WebClerk API) needs to persist across sessions, run it in a separate process. The MCP server itself is spawned per-session by Claude Code, but the backend it talks to must already be running.

7. **On failure, verify in order:** registration → connection → round-trip → session lifecycle. No MCP log file exists to check (as of 2026-06-24).
