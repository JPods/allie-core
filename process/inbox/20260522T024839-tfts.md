  - try: Keep "Export Routes" as a standalone console command to generate route.json
    result: failed conceptually — Build already called generate_feature_json →
            TripPlanner.generate → route.json; Export Routes duplicated that work and
            implied route.json required a separate user action
    revealed: the authority chain (Noelle certifies → Natalie plans → Nora travels)
              was already complete inside Build; a separate export step contradicted it
              and created a false dependency in user mental models

  - try: Revert "Journey" to "Trip"; remove "Export Routes"
    result: succeeded — one vocabulary (Station / Map / Trip / Maneuver) consistent
            across all files; Build is the sole trigger for route.json; error messages
            now say "run Build first" not "run Export Routes first"
    revealed: the right test for a new term is: does it add precision that the old
              term cannot carry? "Journey" and "Trip" meant the same thing; "Trip"
              was already in all the file names, schema fields, and customer language

principle: Vocabulary is load-bearing. Add a new term only if it carries a meaning the
           existing vocabulary cannot express. When customer-facing language (trip.json,
           trip_id, "customers take trips") and internal language diverge, the
           customer-facing word wins — it is the one that must survive contact with the
           real world. The same applies to export commands: if Build already produces the
           artifact, a separate export command is not a convenience — it is a
           contradiction.

definitions_clarified:
  Station: a physical stop. Declared once as a formation template (noelle_features.json).
           Placed in the model by a student. Noelle extracts its routing behaviors into
           {model}.feature.json on every Build. A station has connection points (CPs),
           a platform, and internal guideway segments (gw_platform, gw_uturn_*, etc.).
           Station geometry is authoritative — all routing facts are derived from it.

  Map:     the network. Produced by Build as {model}.followme.json (topology — what
           lines exist and how they connect) and {model}.map.json (agent-facing — station
           positions, platform coordinates, segment lengths in mm). The Map is
           regenerated on every Build. It is Noelle's certified declaration of what is
           physically present. Natalie reads the Map; she does not write it.

  Trip:    one Nora executing a Route from origin to destination. Natalie authorizes a
           Trip by assigning a Route (an O→D segment sequence from route.json) to a
           specific Nora. A Trip decomposes into Maneuvers — one per CP-to-CP segment.
           Natalie dispatches Maneuvers to Nora one at a time; Nora signals arrival;
           Natalie dispatches the next. trip_id is unique per Trip for audit and replay.
           Customers take Trips. Nora executes Trips. Natalie authorizes Trips.
           This is the word in trip.json, trip_id, trip_lines, and the Route-Time GUI.

why_journey_failed:
  "Journey" was introduced (2026-05-20) to distinguish "one Nora executing a Route"
  from "a Route in route.json". The distinction is real. But:
  1. Every existing file already used "trip" for this concept (trip.json, trip_id).
  2. Customers say "I want to take a trip to the station" — not "I want to take a journey".
  3. BFS removed the need for pre-certified routes; Natalie now builds the Trip at
     dispatch time. The distinction between "certified Route" and "executing Journey"
     collapsed — Natalie holds both simultaneously.
  4. Trip is shorter, already in use, and unambiguous in context.
