principle: THE COMPLETE RULE SET FOR ARC TRACKS IN NEW STATIONS AND TRAFFIC CIRCLES:

  1. SOURCE HIERARCHY: beam_path > math synthesis > BLOCK.
     beam_path attribute on JPods Guideway groups is set at Build time from lines.json
     math. It persists. It is not wiped by PathJSON.export. It is the authoritative
     source for arc geometry at animation time. Never use map.json for arc tracks.

  2. UTURN GEOMETRY: radius=1750mm (CENTERLINE_R_G), sweep=180°, diameter=3500mm.
     Validate any 2-pt chord before synthesis: chord must ≈ 3500mm ± 200mm.
     If chord << 3500mm it is a map.json scan artifact (physical beam stub position).
     Skip synthesis and require Build. Do not degrade gracefully.

  3. RING ARC GEOMETRY (traffic circles):
     a. All ring arcs — N_N long arcs (gw_c_0_0) AND N_M connectors (gw_c_0_1) —
        lie on the SAME concentric circle. Same center, same radius.
     b. N_M connector endpoints come from adjacent N_N arc endpoints, NOT from
        scan chord data. Scan chord endpoints are at the beam edge or wrong radius.
     c. In generate_geometry_json: no rail-offset subtraction. outside_r from N_N
        arc fit IS the vehicle-path centerline radius. Use it directly.
     d. Synthesis radius for N_M chords in synthesize_arc_pts_from_math!: use the
        N_N anchor radius (radius from _fit_circle_xy), not the chord's own radius
        (which reflects wrong scan endpoint positions).

  4. CCW ENFORCEMENT: all ring arcs must be stored CCW before build_maneuvers_from_ids.
     Check: cross_z = tx*(center.y - pa.y) - ty*(center.x - pa.x) where tx,ty =
     direction of first segment. positive = CCW; negative → reverse pts.
     Enforce at the lookup layer in enforce_ccw_ring_arcs!, not at maneuver layer.

  5. RING DIAMETER: geometry.json ring radius must match the physical template spec,
     not the scan measurement. Scan over-reads by ~250mm (picks up beam edge).
     traffic_circle7 canonical radius: 7500mm (15m centerline diameter).
     When building new traffic circles: declare the canonical radius in lines.json
     or constants; validate geometry.json against it after generate_geometry_json.

  6. MANDATE GUARD: block animation — do not degrade — if any arc track still has
     2 pts after beam_path upgrade + synthesis. Direct user to run Build.
     "Run Build to regenerate beam_path" is the correct resolution, not workarounds.

  7. PHANTOM ARC ENTRIES — false mandate guard violations:
     Two sources produce 2-pt ARC_TRACK_RE lookup entries that are NOT real unbuilt
     tracks and must NOT trigger the guard:
       a. map.json scan artifacts: a station's map.json may contain stale gw_c_*
          entries from a prior buggy scan even when the formation has no such tracks
          (e.g., station_parking has no gw_c_* but s002 map.json has them from a
          donut-solid edge scan).
       b. Hidden/disconnected Guideway groups: a station may have a hidden set of
          duplicate guideway groups with arc connection_ids but no beam_path. These
          leak 2-pt entries into the lookup via map.json gap-fill.
     Fix: in upgrade_arcs_from_beam_path!, collect real arc cids from VISIBLE
     Guideway groups only (hidden = not active network). After scan, purge any
     2-pt ARC_TRACK_RE lookup entry with no visible group. Real unbuilt visible
     arc tracks are kept and still trigger the guard correctly.
