              eff_xf = world_xf * inner_xf to transform geometry.json pts_mm
              to world coordinates. inner_xf loaded via
              JPods::PathJSON.load_extracted_formation_xf(formation_id).
    result:   traffic_circle7 drew correctly (0 offset). station_thru_dip drew

              far from the station geometry.
    revealed: inner_xf is not zero for all templates. It was captured during
              populate_from_open_template in a network-model context where the
              formation component had a non-zero world position. For
              station_thru_dip the saved inner_xf had a large Y translation.
              traffic_circle7 worked by accident — its inner_xf happened to
              be near-zero.

  - try:      Removed inner_xf multiplication entirely. Set eff_xf = world_xf.
    result:   All templates draw correctly. station_thru_dip: 19 tracks on
              the station geometry with no offset.
    revealed: geometry.json pts_mm are stored in formation-component-local
              coordinates. world_xf (the instance transformation of the
              formation component in the template model) is the only transform
              needed. inner_xf is a network-export artifact — it records the
              offset of the formation component when it was placed as an
              instance in a full network model, not in the template model.skp.

principle: geometry.json pts_mm are in formation-component-local coords.
           To display them in a template model.skp, apply world_xf only.
           inner_xf is valid for network-model export (placed instances at
           non-zero world positions) but must never be applied when reading
           geometry.json for template-local display. The two contexts share
           the same file but require different transforms.
