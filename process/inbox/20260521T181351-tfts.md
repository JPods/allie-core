# TFTS — 2026-05-21T18:13:51Z

problem: Station template line.json startPoint/endPoint extracted via bbox min/max were wrong — every endpoint was "unmatched" at 55mm tolerance because bbox extents are the physical footprint of structural geometry, not the routing connection points.

fault_ref: 20260521T170926

arc:
  - try: Bbox centerline — collect all edge/face vertices, take min/max along dominant axis
    result: failed — all endpoints unmatched; 0mm matches across all 5 templates at 55mm tolerance
    revealed: bbox of a FollowMe-extruded track component includes beam faces, profile curves, and structural elements that extend well beyond the centerline termini; min/max is never the routing connection point

  - try: 4-endpoint min-gap check across all combinations (ep→sp, ep→ep, sp→sp, sp→ep)
    result: partially reduced false positives but still large gaps for curved/ramped tracks
    revealed: curved uturns and ramps have bbox corners that don't correspond to any routing endpoint; the connection point is inside the bbox, not at its edge

  - try: Spatial proximity match — flag any endpoint with no neighbor within 55mm
    result: 100% unmatched across all templates; every endpoint flagged
    revealed: confirmed that bbox endpoints are nowhere near actual connection points for complex station geometry

  - try: Read ene_railroad plugin to understand how it finds track endpoints
    result: decisive insight — ene_railroad does NOT use edge topology or bbox; it stores explicit control points: controls[0]=start_pt, controls[1]=end_pt, controls[2]=start_vec, controls[3]=end_vec; connection matching uses exact point equality, not tolerance
    revealed: the correct approach is to extract or store actual centerline endpoints, not derive them from extruded geometry bboxes

  - try: Valence-1 vertex centroid clustering (implementing)
    result: pending
    revealed: for FollowMe-extruded tracks, all profile-perimeter vertices at each end of the extrusion have valence=1 (connected to only one edge — the profile loop edge); centroid of the valence-1 cluster at each end gives the actual centerline terminus

principle: In SketchUp, extruded geometry (FollowMe, Push/Pull) buries the centerline inside structural faces. The bbox of the result is the structural footprint, not the path endpoints. To recover actual routing connection points from extruded geometry: (1) find valence-1 vertices (edge endpoints connected to exactly one edge), (2) cluster them by position (two clusters = two ends of the extrusion), (3) take the centroid of each cluster. This gives the actual entry/exit point of the track centerline. ene_railroad avoids this entirely by storing explicit control points — JPods station templates should do the same once connection points are confirmed.

domain: SU

why_it_was_hard:
  1. The bbox approach gave plausible-looking numbers (correct lengths) so we didn't immediately see that the endpoints were wrong.
  2. Connectivity checks kept producing false positives, but the failures looked like tolerance issues, not fundamental method failures.
  3. The 100%-unmatched result was the diagnostic that broke the assumption — if even 55mm can't match any endpoints, the method is wrong, not the tolerance.
  4. ene_railroad's source was the reference that made the solution obvious: don't derive endpoints from geometry; compute or store them from the centerline path.
  5. SU edges are the primitive — faces and bboxes are downstream. Connection points live on edges, not on bboxes.
