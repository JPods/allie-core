principle: designer.tracks in lines.json declares the CCW successor order.
           SketchUp geometry is directionless — a line is a line. The designer
           draws from either end; pts[0] and pts[-1] are arbitrary relative to
           routing intent.

           Compute is responsible for normalizing pts_mm direction to match
           the declared successor order before writing lines.computed.json.
           This is not a geometry defect — it is Compute's job.

           Correct Compute pipeline order:
             1. Extract pts from geometry (PathJSON)
             2. Apply cp_marker frame transform
             3. Normalize pts direction per designer.tracks successors  ← here
             4. Pin cp_marker outer endpoints (gw_cp_in/out)
             5. Gap continuity check and snap
             6. Write lines.computed.json

           Direction normalization must run before gap snapping and cp_marker
           pinning so those steps operate on correctly-oriented pts. Running
           after produces confusing log numbers and incorrect snaps.

           The normalization reads all chain types (landing, exit, pass,
           originating) and designer.tracks successors. For each pair (A→B),
           if dist(A.last, B.last) < dist(A.last, B.first) by >50mm, B is
           reversed — flip B.pts_mm. Processes in chain order so each
           correction uses an already-corrected predecessor.
