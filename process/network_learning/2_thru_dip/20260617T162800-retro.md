## Root Cause (from post-run data analysis)

**The coordinate system fix was correct and complete.** World positions in network.json are right — inst_xf * inner_xf applied correctly. The problem is not position. It is arc density in the template source.

**Chain of cause:**

1. `generate_network_json` reads from `templates/track_formations/station_thru_dip/lines.computed.json` and applies inst_xf * inner_xf → correct world pts.

2. `lines.computed.json` has only 2 pts for `gw_cp_in_*`, `gw_cp_out_*`, `gw_cp_in_lead_*`, `gw_cp_out_lead_*`. Correctly has 25 pts for `gw_lift_in`, 17 for `gw_platform_in`, 7 for `gw_uturn_0/1`.

3. The `_lead` tracks are curved approach S-curves. The Compute step that last generated `lines.computed.json` only captured their start/end endpoints — missed the intermediate arc tessellation for these specific tracks.

4. `path.json` has 0 pts for all gw_cp tracks — the path.json pipeline never populated them. So network.json wins (gw_ priority) and animation follows 2-pt chords.

5. Show Track reads arc tessellation from the live model (all edge segment vertices) → draws the full curve. Animation follows 2-pt chord → cuts the corner → sharp heading change at both ends of each lead track.

**Not a code bug.** The geometry extraction pipeline (`write_lines_computed_from_geometry`, triggered by the Compute button) did not capture intermediate arc pts for gw_cp tracks when `lines.computed.json` was last generated. Fix is at the template level: re-run Compute on `station_thru_dip` with correct arc tessellation for all gw_cp tracks.
