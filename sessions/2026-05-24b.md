### CP hub vertex detection

The cp component in station templates has 4 edges: [177mm, 222mm, 1750mm, 1750mm].
- 222mm = outward tangent indicator
- 1750mm × 2 = one edge to each rail (half of 3500mm DUAL_TRACK_SPACING)
- Hub vertex = vertex where all three key edges meet = CP center

Earlier fix attempt used `rv.position == tv.position` to find the shared vertex.
This was unreliable — SketchUp Vertex objects for different edge endpoints can be
at the same position but not pass equality via Point3d comparison.

**Fix:** vertex-degree counting using Ruby object identity:
```ruby
vertex_count = Hash.new(0)
(rail_edges + [tang_edge]).each { |e| e.vertices.each { |v| vertex_count[v] += 1 } }
hub_vertex = vertex_count.max_by { |_, cnt| cnt }&.first
```
The hub vertex is the one that appears in the most key edges (3 = tang + 2 rails).

Also resolved a long-standing mystery: SketchUp was outputting WARN messages in an
old format despite the file having new code. `source_location` confirmed correct file
and line. After `load Sketchup.find_support_file(...)` the new code ran. The old
WARN output was from before the explicit reload.

**Result:** No WARN fallback for any of the 4 station types.

### BEAM_DEPTH/2 for traffic circle

All connections were straight but TC end was 0.25m lower than non-TC ends.

`compute_anchor_zs` in `jpod_network.rb` used `+ BEAM_DEPTH/2` for TC and
`+ BEAM_DEPTH` for others. This was confirmed flush on 2026-05-14 with old
template geometry. With new cp instances (commit 37b7bd9), TC hub vertex Z =
7.75m — 0.25m lower than old derived reference. BEAM_DEPTH/2 = 0.25m no longer
correct; need full 0.5m.

**Fix:** removed is_tc distinction from compute_anchor_zs:
```ruby
from_z = from_cp[:center].z + Constants::BEAM_DEPTH
to_z   = to_cp[:center].z   + Constants::BEAM_DEPTH
```

**Result:** All 4 inter-station connections flat at 8.25m. User: "Looks good."

## Key decisions

**vertex-degree counting over position comparison** — When finding the shared vertex
of multiple edges in SketchUp Ruby, use Ruby object identity (Hash keyed on Vertex
object). Point3d == is tolerance-based and fails across separate Vertex instances.

**Remove is_tc BEAM_DEPTH/2** — The TC special case was compensating for a derived
reference Z that no longer exists. With explicit hub vertex detection, all station
types use the same offset: hub_z + BEAM_DEPTH = beam top.

## Lessons for Allie

1. **SketchUp Vertex identity vs position equality** — Two Vertex objects from
   different edges may report the same position but fail Point3d == comparison.
   Use the Vertex object itself as a Hash key when counting shared vertices.
   vertex_count[v] += 1 using `e.vertices.each` gives reliable hub vertex detection.
   Domain: [SketchUp]

2. **CP anchor Z: remove derivative compensation after explicit placement** —
   When a cp component is explicitly placed in a template with origin at a known
   reference, all compensation offsets (like BEAM_DEPTH/2 for TC) that were
   adjusting for a different derived reference should be removed. The explicit
   placement IS the reference. Domain: [SketchUp]

3. **Verify source_location before diagnosing a load mystery** — SketchUp appeared
   to be loading old code. `source_location` confirmed the correct file and line.
   The actual cause: the explicit `load` hadn't been run since the last save.
   Run source_location first when WARN output doesn't match expected code format.
   Domain: [SketchUp]
