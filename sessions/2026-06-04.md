## Scars This Session

**CP numbering mismatch is invisible until gap-check** — The traffic_circle7 gw_cp_in_N
tags were at wrong CP positions (systematic rotational offset). The model looked correct in
the viewport — all components in the right places visually. The only signal was the junction
gap check showing 11,042mm and 17,254mm gaps (not random noise — exact geometrically
predictable distances). The gap sizes are the diagnosis. Cost: ~3 passes and an axis
unification detour. Rule: when junction gaps are suspiciously exact and large (not the usual
1–40mm alignment errors), the tracks are at the right positions but wrong tags — look at
Entity Info on the components and compare against cp.json.

**Axis unification changed absolute world positions** — The Ruby bake-rotation script
preserved all relative positions but shifted the model's world coordinates. The cp.json
gate positions changed after re-running Extract Template. This is acceptable for templates
(absolute world position doesn't matter, only relative internal geometry matters) but would
be catastrophic in a network model. Rule: axis unification scripts are template-only
operations. Never run against a network model.
