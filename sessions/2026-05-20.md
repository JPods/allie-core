# Session Log — 2026-05-20 (Session 2)

## Goal

Fix two bugs reported after the pipeline was proven working: (1) Noelle reporting false approach curve BLOCK violations at ~1.2m radius on visually-correct guideways; (2) waypoints between S048 and S050 being dropped during Build.

## Accomplished

- **Investigated approach curve false violations** — 5 BLOCK violations at 1.2m radius on S048, S050, S051 inter-station connections. Visually all were 3.5m+ radius.
- **Attempt 1 (failed):** Added 0.3m minimum chord filter before circumradius calculation. Violations moved from 0.9m to 1.2m — still present.
- **Attempt 2 (failed):** Added `APPROACH_SKIP_DEPTH = 2.0m` constant and skip zone in `check_approach_curves`. Violations still at 1.2m after multiple SketchUp restarts.
- **Root cause found:** `check_approach_curves` was calling `vehicle_path_for`, which stitches station-internal U-turn geometry onto the path for animation continuity. The U-turns have genuine ~1.2m radius in XY when viewed from circumradius measurement. The inter-station guideway is irrelevant once the station path is prepended.
- **Fix applied:** Changed `vehicle_path_for` → `base_vehicle_path_for` in `noelle.rb:871`. `base_vehicle_path_for` reads only the guideway's own `beam_path` attribute — no stitching.
- **Result:** "Noelle approach curves: all guideways approved (min radius >= 8.0 m)." All 5 violations eliminated.
- **TFTS written:** `~/Allie/process/inbox/20260520T232624-tfts.md` — 3-attempt arc documented.

- **Investigated waypoint dropping** — Build log showed `via_markers=[1,2]` on the S048-S050 pair instead of `[1,2,3]`. Guideways built straight without waypoints; columns appeared at waypoint positions separately.
- **Initial misdirection:** Attempted to fix by mirroring via_markers from forward to reverse direction in feature.json. User flagged: wrong approach — "We corrected this earlier today."
- **Root cause found:** `generate_feature_json` was creating TWO cp_ entries for each bidirectional connection (one per direction). `build_from_config` called `build_segment` twice for the same physical pair. `build_segment` with `replace:true` erases BOTH `seg_fwd_id` and `seg_rev_id` guideways before rebuilding — the second call always overwrites the first, stamping via_markers from the SECOND cp_ entry (which had `[]`).
- **Fix applied:** `generate_feature_json` now checks if the reverse cp_ key already exists in `conns_out`. If yes, nest the current seg_ entry into the existing entry. ONE cp_ entry per physical pair, containing both directional seg_ sub-entries.
- **Result:** feature.json dropped from 6 cp_ entries to 3. Trip S048→S050: 465m → 556m (waypoints being routed). `via_markers: [1,2,3]` on the S048-S050 pair.
- **TFTS written and amended:** `~/Allie/process/inbox/20260520T234607-tfts.md` — complete arc + warning about future single-guideway (one-way) connections requiring merge logic revisit before implementing.
- **Committed:** Both TFTS files committed to `~/Allie`. Plugin changes committed as `14f743f` on `su_jpods_claude` branch.
