# Session — 2026-06-11 (session 3)

SketchUp Plugin — JPods_station_parking platform_shuffle test: CCW violation, direct-to-final-slot advance

## What was accomplished

Fixed two remaining failures in the platform_shuffle test for JPods_station_parking. The fixes from session 2 (shuffle_parked lifecycle, hold_loop_parked transition) were working correctly — pods were advancing — but with two new failures visible in the log.

### Bug 1 — CCW mandate violation: pods traveling backward through other pods

**Root cause**: `_platform_pts_entry_first` uses pod position (`pod_pos`) as its orientation fallback when `gw_platform_parking` is absent from the lookup cache. JPods_station_parking has no `gw_platform_parking` track — it uses `gw_platform_in2` as the final landing track before `gw_platform`. Once a pod advances past the midpoint of `gw_platform`, `d_rev < d_fwd` and the fallback reverses the pts array. `advance_pod_slot` then clips a path from the pod toward the entry end (backward) instead of the exit end.

This is why higher-slot advances (ps6→ps9) produced CCW violations: ps7, ps8, ps9 are all in the lift section beyond the midpoint; the pod was dispatched in the wrong direction.

**Fix**: Replace pod_pos fallback with Sally ps1 slot_position anchor (`jpod_vehicle_anim.rb`, `_platform_pts_entry_first`).

Sally.init_from_model uses the same raw pts direction when `gw_platform_parking` is absent (no reversal). ps1's 3D coordinate was computed on that same un-reversed pts. Anchoring on ps1 guarantees that `advance_pod_slot` uses the identical orientation frame Sally used when building `@@slot_positions`. The pod_pos path is kept as a last fallback only when Sally is not loaded.

### Bug 2 — Spot-by-spot advance (7 ticks per pod) instead of direct-to-final

**Root cause**: `_advance_platform_queues` was scanning for the topmost empty slot with an occupied slot directly behind it — advancing one slot per tick. With cap=9 and a pod at ps2, that's 7 ticks (7 seconds) to reach ps9.

Bill's intent: when nora_001 departs for its loop, Sally dispatches each parked pod directly to its compacted final slot in one command.

**Fix**: Rewrite `_advance_platform_queues` in `jpod_sally.rb` to compute compacted targets directly.

Algorithm:
- Sort occupied pods DESCENDING by current slot
- Assign targets: rank 0 (highest current) → cap, rank 1 → cap-1, …
- Advance ONE pod per tick (break after first); ~1s stagger between pods

Highest-slot-first ordering ensures the pod closest to the exit clears the track before the pod behind it starts moving. The 1s stagger (from break + PLATFORM_ACTIVE_TICKS cadence) provides enough separation at pod speed (~5-8 m/s) to never create a personal space violation.

Result: with 2 pods and cap=9 — tick 1 dispatches highest pod to ps9, tick 2 dispatches remaining pod to ps8.

### Bug 3 — "advance_pod_slot: NORA_0001 not in active pods" log spam

**Root cause**: After a pod exits via originating chain, `@@pods.reject!` removes it from the animation pool. But in some edge cases (compact_parked fallback, timing edge) Sally's slot registry was not cleared. `_advance_platform_queues` fired every second and tried to advance the ghost entry.

**Fix**: In `advance_pod_slot`, call `Sally.release_slot(sid, nora_id)` when the pod is not found in @@pods. Defensive cleanup at the point of detection.

### Bug 4 — compact_platform_idle crash: model.entities nil

Same nil guard added to `confirm_slots` in session 2. `compact_platform_idle` had an unguarded `model.entities.each` at the idle_reserve scan. Added `entities = model&.entities; return unless entities`.

## Principles confirmed

1. **Platform pts orientation must match Sally.init_from_model's frame.** `_platform_pts_entry_first` and `init_from_model` must use the same orientation logic for any given template. When `gw_platform_parking` is absent, both leave pts un-reversed. ps1's slot_position (computed by Sally on that direction) is the correct anchor — pod position is not.

2. **Direct-to-final compaction, highest slot first.** When multiple pods need to compact toward the exit end, the pod closest to exit must advance first. This clears the track. One dispatch per 1s tick provides a natural stagger that prevents personal space violations without inter-pod coordination.

3. **Stale registry entries must be cleaned up at the detection site.** When `advance_pod_slot` finds a pod missing from @@pods, it calls `Sally.release_slot`. The registry and the animation pool must agree — catch divergence at the point where it's observable.

## Lessons for Allie

1. **Pod position is an unreliable orientation anchor past the platform midpoint** — `_platform_pts_entry_first` used pod_pos as fallback: once pod advances beyond the halfway mark, d_rev < d_fwd and the orientation reverses. The correct anchor is Sally's ps1 slot_position — it was computed on the same raw pts direction Sally uses internally. Check: any template without `gw_platform_parking` that expects advances past the midpoint must have Sally loaded (ps1 available) or the fallback will misbehave. [SketchUp]

2. **Direct compaction order: highest slot first** — Pods compact toward the exit end. The highest-slot pod (closest to exit) must move first. If the lowest-slot pod goes first, it may travel through other pods. Sort descending, advance one per tick. This gives O(n) ticks with natural spacing at pod speed. [SketchUp]

3. **Sally slot registry and @@pods must agree; fix at detection** — When @@pods doesn't have a pod but Sally's registry does, the registry entry is stale. `advance_pod_slot` is the right place to clean it: it has both the sid and nora_id, and the fix (release_slot) is one line. Do not let the registry diverge silently. [SketchUp]
