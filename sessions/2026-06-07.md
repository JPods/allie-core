## Lessons for Allie

1. **`Geom::ORIGIN` is not accessible inside lambdas in SketchUp Ruby** — Use
   `Geom::Point3d.new(0,0,0)` instead. The constant lookup `Geom::ORIGIN` fails inside
   closures/lambdas in the JPods module context. Domain: [SketchUp]

2. **ene_railroad solid geometry has no standalone edges** — When FollowMe extrudes a
   cross-section, the resulting solid's edges are only accessible via `face.outer_loop.edges`,
   not as standalone `Sketchup::Edge` entities iterated directly. Any indicator geometry meant
   to be found by code must be a standalone edge NOT welded to ene_railroad face geometry.
   Domain: [SketchUp]

3. **When SketchUp makes a component from drawn geometry, the component's local origin may
   not align with the edge endpoint** — Do not require proximity to `Geom::ORIGIN` for
   externally placed components. Use the component's world insertion point
   (`Geom::Point3d.new(0,0,0).transform(world_xf)`) to determine which end of the edge
   is the "from" end. Domain: [SketchUp]

4. **Diagnostic scaffolding pattern** — When a search fails silently (no output, no error),
   the issue is often in the diagnostic condition, not the search itself. Check: does `_diag`
   use the RIGHT attribute (layer name on the instance, not definition name)? Both can be
   different. Domain: [SketchUp]

5. **Make the indicator visible to designers, not hidden in component geometry** — Placing
   the vector_in component outside the track group but inside the bounding envelope solves
   two problems: designers can see and orient it; code can find it via a model-level tag scan
   rather than deep geometry traversal. Axiom 10 (explicit model datum) requires the datum
   to be inspectable. Domain: [SketchUp]
