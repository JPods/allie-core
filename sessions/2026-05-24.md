in SketchUp's rendering pass and z-fights with coincident faces in other groups/components.

position = z-fighting. Erasing removes the beam's face entirely.

## Lessons for Allie

1. **Erase ≠ Hide for faces at junctions** — hiding a face leaves it in the geometry.
   At a gate junction where two separate SketchUp groups/components both have a face at
   the same XY position, those faces z-fight in the viewport regardless of hidden state.
   Erasing removes the face. For beam caps at gate connections, always erase.
   Domain: [SketchUp]

2. **Bounding box corners projected ≠ gate outer face** — computing cp gate position by
   projecting bounding box corners onto the outward tangent gives inconsistent results
   (111mm vs. 296mm for different station types). The bounding box includes ALL geometry
   in the cp component definition, including decorative indicators that may extend past
   the actual gate face. Don't use bounding box extents to determine gate position.
   If the cp component was placed with origin at gate, use origin. Check origin= in logs.
   Domain: [SketchUp]

3. **stub_pair_length_mm is a constant that can get stale** — it's hardcoded in
   generate_map_json. When the user changes template geometry (stubs from 2m to 9m to 5m),
   this constant must be updated manually. Consider reading stub length from CP data or
   line.json rather than hardcoding. For now: 5000.0. Flag if user changes stubs again.
   Domain: [SketchUp]

4. **When a fix requires reading multiple files (map.json, path.json, feature.json),
   the fix must live in ALL of them** — per TFTS 20260523. cap_pt snapping in map.json
   wasn't enough because path.json didn't have it. The ene_railroad open-tube approach
   (no caps) avoids this multi-file problem entirely by not creating the artifact.
   Domain: [SketchUp]
