# Retrospection — 2026-06-17

## Session: traffic_circle7 Transit Test — three bugs, one principle

### What Was Done

Four fixes to complete the transit_test for traffic_circle7:

1. **Run button inert** — `JSON.stringify(formation)` in `renderModelTests` embedded
   double-quoted strings inside a double-quoted HTML onclick attribute. Browser ended the
   attribute at the first internal `"`. Fix: single-quoted JS template literals.

2. **Vehicles moving CW** — pass_chains list track names in CCW traversal order, but
   some `pts_mm` arrays are stored in the reverse direction (e.g. `gw_c_1_1` goes
   right→left in pts_mm but the CCW path needs left→right). Transit intercept was
   concatenating pts in stored order. Fix: before stitching each track, compare
   `last.distance(pts.first)` vs `last.distance(pts.last)` and reverse when the far
   end is closer to the current endpoint.

3. **Vehicles not following Show Track lines** — `show_track_overlay` and the console's
   `template_lookup` builder both search for the formation component's world transform, but
   used different search logic. show_track_overlay checks both `ComponentInstance` and
   `Group`, with a Pass 2 fallback scanning for gw_*-tagged children. The console builder
   only checked `ComponentInstance` with no fallback. If the formation is a Group (or
   unnamed ComponentInstance), the console used identity while Show Tracks used the real
   transform. Fix: console builder now mirrors show_track_overlay exactly.

4. **model.network.json created by Build on template** — `generate_network_json` in
   noelle.rb had no template guard. Fix: return early if
   `model.path.include?('track_formations')`. Template models only produce
   lines.json + lines.computed.json.

**Result:** Bill: "Looks great." Transit Test PASS.

## Lessons for Allie

1. **pts_mm direction is not guaranteed by pass_chains** — pass_chains list track names
   in correct traversal order but do NOT guarantee that pts_mm is stored in that direction.
   Any code that builds a pts path from a track list must check which end of each track
   connects to the current path endpoint before appending. The canonical pattern:
   ```ruby
   tk_pts = tk_pts.reverse if last.distance(tk_pts.last) < last.distance(tk_pts.first)
   all_pts.concat(tk_pts[1..])
   ```
   This will silently fix reversed arcs without needing to audit each track's pts_mm
   direction in lines.computed.json. [SketchUp]

2. **Any function that reads formation instance transform must use the same search** —
   Two functions searching for "the formation component" in the same model will diverge
   silently if they use different search strategies. The canonical two-pass search:
   Pass 1: find entity with `JPods.model_id == formation_id` OR `definition.name == formation_id`
   (check both `ComponentInstance` and `Group`).
   Pass 2: if still identity, find entity whose direct children include gw_*-tagged entities.
   This pattern lives in `show_track_overlay`. Any other code that needs the formation
   transform must duplicate it exactly. [SketchUp]

3. **Template guard belongs in generate_network_json, not at the call site** — Callers
   (Build, Validate, standalone scripts) should not need to know the two-file rule. The guard
   inside `generate_network_json` is the right place — it enforces the invariant regardless
   of which entry point triggers it. Same principle applies to any function that produces
   model.*.json output. [SketchUp]

4. **HTML attribute quoting: JSON.stringify produces double quotes** — `JSON.stringify(str)`
   always returns `"value"` (with double quotes). Embedding that in an onclick="..." attribute
   silently breaks the handler — the browser closes the attribute at the first inner `"`.
   For onclick attributes, always use single-quoted JS string literals:
   `` `onclick="fn('${val}')"` `` not `` `onclick="fn(${JSON.stringify(val)})"` `` [SketchUp]

## Scars

**pts_mm direction** — An hour tracing CCW/CW behavior before realizing the issue was
not in the pass_chains logic but in the direction the pts are stored. The distance-to-endpoint
check is simple but non-obvious — it requires knowing that track pts_mm can be stored
"backward" relative to traffic flow.

**transform search divergence** — Show Tracks worked. Vehicle animation worked. They just
didn't agree on WHERE to draw. Both functions used "the same" source data (lines.computed.json)
but retrieved different transforms. Silent coordinate mismatch is the hardest class of bug.
Rule: any function that computes a world transform from template-local pts must reference
ONE canonical search function, not duplicate the logic.
