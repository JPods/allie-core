## Session 1: Waypoint UX and Live Bezier

### What Changed

**Cursor state machine (onSetCursor)**
- `@dragging_marker_num` set → cursor 648 (4-arrow move)
- `@hover_marker` (within 2 m of a marker in XY) → cursor 643 (hand/grab)
- Gold/Cyan with `@pending_via_pts.any?` → cursor 671 (crosshair — precise placement)
- Gold/Cyan, no pending → cursor 280 (pencil — click a gate or press W)
- Idle → cursor 671 (crosshair)

`@hover_marker` is tracked each `onMouseMove` by scanning the cached `@markers` array
for the nearest point within 2 m (XY distance only). Cheap enough to call every frame.

**Live Bezier during marker drag**

Root cause: the draw method for the cyan edit state read `@edit_preview_pts`, which is
only updated when the cursor moves outside a drag. During a drag, `onMouseMove` takes
the early-return drag branch and never reaches the `@edit_cp` preview block.

Fix: replaced `@edit_preview_pts` with `d[:paths]` in the draw method.
`d[:paths]` is rebuilt by `rebuild_draft_paths` on every drag tick
(via `update_draft_via_pt` → `rebuild_draft_paths`). Drawing from `d[:paths]` means
the bezier follows the marker in real time, frame by frame.

The same change applies to `d[:center_pts]` for the dashed centerline — it was already
using the draft directly, so that was already correct.

**`@@draft_connections` as single source of truth**

All rendering reads from `@@draft_connections`, not from JSON or `@planned_path_entries`.
- `visible_draft_connections` — all drafts except the one being actively built/edited
- `draft_for_cp(cp)` — the draft owning a given CP
- `rebuild_draft_paths(idx)` — rebuilds both `d[:paths]` (dual track) and `d[:center_pts]` (dashed center)

JSON is write-only from the draw perspective. The disk file is only read on tool
re-activation (`restore_draft_connections`).

## Session 2: Build Pipeline Crash

### Error

```
Build segment seg_S012_cp1_S044_cp0 error: Cannot convert argument to Geom::Vector3d
```

### Root Cause — Critical SketchUp Ruby API Constraint

**SketchUp's `Geom::Vector3d` does NOT support the `*` operator with a Float scalar.**

```ruby
vec.normalize * 2.5   # → ArgumentError: Cannot convert argument to Geom::Vector3d
```

The correct form is always explicit component multiplication:

```ruby
n = vec.normalize
Geom::Vector3d.new(n.x * 2.5, n.y * 2.5, n.z * 2.5)
```

This is a SketchUp-specific constraint — it does NOT work the way Ruby's Numeric `*`
usually does with custom classes. There is no `coerce` method registered on Vector3d.

### Where It Appeared (all fixed)

| File | Method | Line (approx) | Expression |
|------|--------|----------------|-----------|
| `jpod_connect_tool.rb` | `offset_path` | ~1113 | `avg.normalize * miter` |
| `jpod_connect_tool.rb` | `bezier_pts_via` | ~1332 | `unit * tangents[i].dot(unit)` |
| `jpod_connect_tool.rb` | `bezier_pts_via` | ~1337 | `bis.normalize * s` |
| `jpod_network.rb` | bezier method | ~900 | `unit * tangents[i].dot(unit)` |
| `jpod_network.rb` | bezier method | ~905 | `bis.normalize * s` |
| `jpod_network.rb` | `offset_path` | ~961 | `avg.normalize * miter` |
| `jpod_animator.rb` | arrow drawing | ~3404 | `perp_raw.normalize * 0.4.m` |

### Why It Appeared in Multiple Files

`jpod_connect_tool.rb` and `jpod_network.rb` both contain their own copies of:
- A Catmull-Rom / Hermite bezier sampler
- An `offset_path` method that offsets points perpendicular to path direction

When the connect tool's bezier was refactored to use Catmull-Rom with forward-agreement
tangent correction, that math was also present in the build pipeline's own copy.
Fixing one without the other left the build broken.

**Rule going forward: if the same algorithm exists in both files, fix both files.**

### Build Pipeline Flow (for reference)

```
Extensions > JPods > Create > Build
  → jpod_network.rb Network.build_all_segments
    → for each segment in followme.json:
        PathBuilder.build_center_path   (jpod_path_builder.rb)
        Network.offset_path             (jpod_network.rb) ← was crashing
        Network.bezier_pts_via          (jpod_network.rb) ← was crashing
        entities_builder.build_beam     (jpod_entities_builder.rb)
```

## Lessons for Allie

1. **`Vector3d * Float` always crashes in SketchUp Ruby.** Never write `vec * scalar`.
   Always expand to `Geom::Vector3d.new(vec.x * scalar, vec.y * scalar, vec.z * scalar)`.

2. **Algorithm duplication across files = two bug sites.** The bezier and offset_path
   logic exists independently in `jpod_connect_tool.rb` and `jpod_network.rb`.
   Any change to one must be applied to both.

3. **Draw from live state, not from stale instance variables.** When a drag updates
   `@@draft_connections` in real time, the draw method must read from `@@draft_connections`,
   not from instance variables that were set before the drag started.

4. **SketchUp cursor IDs (confirmed working):**
   - 671 = crosshair
   - 280 = pencil
   - 643 = hand/grab (pointer)
   - 648 = 4-arrow move

5. **`onSetCursor` fires frequently.** Cursor state logic must be O(1) — no model scans.
   Precompute `@hover_marker` in `onMouseMove` from the cached `@markers` array.

## Session 3: Clearance Height Design Decision and Risk Documentation

### What Changed

**Root cause of wrong guideway height:** Legacy markers placed at old guideway height
(7.5 m) had their Z stored in the SketchUp group bounding box. `build_segment` passed
that Z into the Bezier spline as `desired_z`, which `apply_vertical_profile` honored
when grade limits permitted. The guideway stayed at 7.5 m above terrain at those
waypoints instead of terrain + 4.6 m.

**Fix:** `jpod_network.rb` line 480 — marker XY is authoritative; Z is zeroed before
PathBuilder runs. PathBuilder terrain-snaps in step 2 and applies CLEARANCE_HEIGHT in
step 3. Stored legacy Z now has zero influence.

**Canonical location for height values:** `jpod_constants.rb`
- `CLEARANCE_HEIGHT = 4.6.m` — single source of truth; change only here
- `WAYPOINT_STANDOFF` — alias for same value; clearer intent at call sites
- Exposed in Network Editor › Constraints panel ("Guideway Clearance Height (above terrain)")
- Safety note added in constants file explaining the 7.5 → 4.6 m change and what is required

### Design Decision — 4.6 m Clearance

**Old design (7.5 m):** Passive safety. Standard semi (4.1 m) and double-decker bus
(4.3 m) pass under with ≥3 m margin. No active detection required. Mirrors AASHTO
highway overpass standard.

**New design (4.6 m):** Active safety required. Bottom of beam ~4.35 m above terrain.
Overheight trucks, raised dump beds, and double-stack flatbed loads enter this envelope.

**Why 4.6 m:** Urban column cost and density. 7.5 m columns are impractical in tight
corridors. The cost and geometry trade is justified, but the safety model changes.

**Bill's comparison:** Roads provide no active protection for pedestrians from vehicles —
separation is the design. JPods at 7.5 m had the same model: height was the protection.
At 4.6 m, height is no longer sufficient. Sensors and defensive pod stop are required.

**Bill's design rule (2026-05-13):** The two go together. Deploying at 4.6 m without
active height sensing and pod defensive stop is not an acceptable interim state.
