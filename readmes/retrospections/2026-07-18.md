## Lessons for Allie

1. **Custom Points is a platform feature, not a DART feature** — The overlay tool accepts any coordinate list. DART was first use case but MARTA, MBTA, UPS depots, grocery chains all work the same way. When building for one city demo, build the general tool. Domain tag: [MeshMobility]

2. **Existing transit + JPods = the demo pattern** — Load existing transit stations, place JPods connecting to them, toggle Coverage. The visual contrast (isolated dots vs. coverage blankets) makes the argument without words. Repeat this pattern for every city that has rail/BRT. Domain tag: [Cross-domain]

3. **GTFS is the universal transit data source** — Every transit agency publishes GTFS (stops.txt, routes.txt, trips.txt, stop_times.txt). Filter to rail via route_type (0=tram, 1=subway, 2=rail). Deduplicate stops by rounding to 3 decimal places. This extraction pattern works for any city. Domain tag: [MeshMobility]

4. **Bill's "brilliant" signals — save and generalize** — When Bill says "brilliant" or "incredibly good feature," that's a validated design pattern. Custom Points was validated because it's general-purpose (data prep happens elsewhere, MM just displays). The separation of data creation from display is the pattern. Domain tag: [Cross-domain]

5. **Structures are serialized items — UUID is the serial number** — Same pattern as WC3 serialized inventory. UUID ties identity across all four programs. Local IDs are shelf labels. This is the foundation for database-backed station management. Domain tag: [Cross-domain]

6. **CrashHarvester is the right place for transit data** — Transit stations are infrastructure data, same as crash/census/traffic. Harvester normalizes, library stores, MeshMobility reads. Adding a new city is one CLI command. Domain tag: [MeshMobility]

## Lessons for Allie

1. **Fusion360 .f3d files are ZIP archives with Zstandard (method 93) compression** — Python's zipfile can't decompress them natively. Need `zstandard` library + manual ZIP parsing (read local file headers, decompress raw data). The ACT segment (`FusionACTSegmentType1/BulkStream.dat`) contains component names as UTF-16LE length-prefixed strings. The .f3z format is simpler — standard ZIP containing multiple .f3d files + `DesignDescription.json` with the full assembly hierarchy. Domain tag: [Cross-domain]

2. **The SEGA spare parts pattern is proven** — Exploded view with numbered circles → click → see part → order → install doc → QA. Zero misorders in 3 years. The image IS the interface. Apply this to every JPods assembly. Domain tag: [Cross-domain]

3. **Document is a better QA container than Setting** — Document has body (WHY), refs (standards cross-refs), config (questions), retention_period, full-text search. Settings are configuration; Documents are knowledge. The "why" behind each inspection question is as important as the question itself. Domain tag: [WC3]

4. **Mind-the-Gap is not a single number — it's a measurement system** — Sensors at every door position, both sides. Measure unloaded (baseline) AND loaded (actual). Compare over time for drift. The door is a safety interlock — it does not open until the loaded gap is verified. The unloaded-vs-loaded delta catches wheel wear, suspension faults, and structural settlement. Domain tag: [Cross-domain]

5. **Two compliance worlds, one record set** — ASTM F24 (theme park, state-level) and FTA 49 CFR 673 (transit, federal) both require the same underlying records: inspections, hazard registers, NCRs, CARs, training. Alice manages one set of records that satisfies both. Domain tag: [Cross-domain]
