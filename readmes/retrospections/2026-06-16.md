### Session 2 (this session): Sequencer count=0 root cause + advance orientation chain

**Root cause chain identified:**
1. `formation_id` attribute never written to SketchUp component instances for thru_dip stations
2. Entity scan in `init_sequencers_from_model` finds entities with structure_id only if formation_id is also present — both must exist for the fallback to fire
3. `template_from_json` loaded but never consumed → sequencer count=0
4. sequencer=0 → `seq_chain_track = ''` → ptrack = st[:parking_track] = 'gw_platform_parking'
5. Slot positions computed on gw_platform_parking with inverted orientation (entry_anchor via gw_platform_in works geometrically but produces ps1 near the exit end in this model)
6. advance_pod_slot clips gw_platform_parking; passes from_pt (middle slot) to `_platform_pts_entry_first`
7. `_platform_pts_entry_first`: pp_ent = gw_platform_parking = SAME track as being oriented → self-referential → d_first == d_last == 0 → no reversal → falls to ps1 anchor
8. ps1 position is near the EXIT end (wrong orientation from step 5) → pts get REVERSED by ps1 anchor
9. All advances end at the exit end of gw_platform_parking

**Fixes applied:**

`jpod_sally.rb` — `init_sequencers_from_model`:
- After entity scan, if count==0 AND template_from_json is non-empty: iterate template_from_json directly and call init_sequencer_for_station for each sid
- This is now the primary path for thru_dip (entity attrs absent by design)

`jpod_vehicle_anim.rb` — `_platform_pts_entry_first`:
- Detect self-referential pp_ent (pp_ent[:pts].equal?(raw_pts) or ==): skip proximity check
- Falls to ps1 anchor correctly when anchoring gw_platform_parking against itself

`jpod_vehicle_anim.rb` — `advance_pod_slot`:
- Pass `entry_anchor = Sally.slot_position(sid, 1)` to `_platform_pts_entry_first` instead of `from_pt`
- ps1 is always at the entry end by definition; from_pt (middle slot) is not a reliable proxy for orientation
- Enhanced diagnostic logging: tracks orientation anchor, pts[0]/pts[-1], tail size, advance size, from/to coordinates

`jpod_sally.rb` — `on_maneuver_complete`:
- Platform-full branch: reset `@@platform_quiet_ticks = 0` when pod must loop again (full platform)
- This ensures `poll_sally_dispatch` fires within 1s to dispatch ps_cap pod and open a slot
- Log includes: pod count vs capacity, dispatch candidate nora_id

## Lessons for Allie

1. **network.json is always authoritative after Build; entity attributes are not** — `formation_id` is never written to thru_dip component instances. `init_sequencers_from_model` must iterate `template_from_json` directly after entity scan if count==0. Never rely on entity attrs as sole source of template identity.
   Domain: [SketchUp]

2. **Self-referential proximity check produces zero signal** — when orienting track T using T's own pts as anchor, d_first == d_last == 0. `_platform_pts_entry_first` must detect and skip this case. Symptom: all advances end at the same wrong endpoint (always the EXIT end).
   Domain: [SketchUp]

3. **Never use the pod's current position to orient the track it is advancing on** — from_pt (middle slot) reversed orientation on gw_platform_parking. Always use ps1 (entry-end definition) as the orientation anchor in `advance_pod_slot`. ps1 is always at the entry end by design; the pod's current position is not.
   Domain: [SketchUp]

4. **The recurring parking bug chain has a single root cause: ptrack** — every session the "wrong slot positions / advance fails / pods at same location" bug traces to the same cause: ptrack is wrong (gw_platform_parking instead of gw_platform). Correct sequencer initialization fixes the entire chain. When this bug appears again, check sequencer count first.
   Domain: [SketchUp]

5. **Platform full → dispatch ps_cap immediately** — when on_maneuver_complete decides to keep a pod looping (platform full), reset @@platform_quiet_ticks=0. This activates the 1-second cycle and fires poll_sally_dispatch within 1 second. Without this, the looping pod must wait up to 4 seconds for the idle cycle.
   Domain: [SketchUp]

## Scars

- **The orientation chain** — three bugs (sequencer init, self-referential anchor, from_pt as proxy) each individually looked correct. The chain only showed as a single symptom: all advances ending at the same wrong endpoint. Root cause was always sequencer=0. Fix one root cause, collapse the whole chain.

- **formation_id never written for thru_dip** — this has been the invisible root cause of multiple sessions of parking debugging. It should be written to entity attrs in the Build step (Noelle), OR the sequencer init should always prefer network.json over entity attrs. The current fix (network.json direct fallback) is the right long-term answer.

### Lessons for Allie

1. **Direct-park is the one hold_loop pattern** — When `to_platform=[]` and the loop ends
   at `gw_platform_parking`, Sally's `on_maneuver_complete` returns `{ action: :land, tracks: [] }`
   and the direct-park code fires immediately. One dispatch, no race, no pre-queued park_man.
   The intersection-approach path (`_final_approach_tracks` + `_enqueue_hold_loop_return_park`)
   was added for complex topologies but proved brittle in practice. Retire it for any template
   where the loop can be routed to end at gw_platform_parking. [SketchUp]

2. **Loop ending at gw_platform_parking ≠ loop being a closed circuit** — station_thru_dip's
   old loop ended at gw_uturn_1 (closed circuit back to loop[0]). New loop ends at
   gw_platform_parking (open route). For target_loops=1 (shuffle test) and target_loops=0
   (first-available-slot), the loop only runs once before direct-park fires, so the route
   does not need to be a closed circuit. Only multi-loop (target_loops > 1) would expose the
   jump from gw_platform_parking back to gw_cp_in_lead_1 — not currently tested or deployed. [SketchUp]

3. **"One function" means same code path, not same geometry** — Bill's directive "there needs
   to be only one sally animation function" means all templates must share the same dispatch
   code path. That path is the direct-park :land with tracks=[]. The geometry (loop tracks)
   differs per template; the code path is identical. When a template uses a different path
   (intersection approach), it's a template data bug, not a code bug. Fix the lines.json. [SketchUp]

4. **Blank station_test_phase + departure_mode is a trap** — When departure_mode=true and
   phase is blank, poll_sally_dispatch's else branch clears phase to '' before the pod
   reaches ps_max. Dwelling loop sees blank phase → _dispatch_station_exit fires at current
   slot → fails → pod erased. Fix: always set shuffle_parked at placement time for departure
   test pods. Blank phase is now only legal for non-template pods in network mode. [SketchUp]
