## Lessons for Allie

1. **Return values from void functions create silent retry loops** — `compact_platform_static` returned nothing; the caller assumed it worked and retried regardless. Any function that may or may not do work must return a count or boolean. [SketchUp]

2. **Total model count vs. per-entity count is a common confusion point** — `existing_count` counted ALL vehicles in the model; `at_station` counts only vehicles at the specific station. Dispatch caps must always be scoped to the relevant unit (station, platform, segment). [Cross-domain]

3. **nil destination crashes at attribute access, not at argument passing** — `place_vehicle_at_platform(model, ..., nil)` accepts nil silently; the crash happens 10 lines later at `destination_platform['id']`. Guard nil at the point of use, not just at the function entry. [SketchUp]

4. **Rescue without erase creates ghosts** — when an exception fires after `place_vehicle` but before attribute setting, the vehicle is in the model with no `vehicle_id`. It blocks future placements, is invisible to all fleet operations, and persists across sessions. The rescue block must erase `inst` before returning false. [SketchUp]

5. **Trace prints that bracket a silent failure are the fastest diagnostic** — two `[PVP-TRACE]` lines (after `place_vehicle` and after `resolve_placement_overlap`) immediately showed the crash was between `placement_ok=true` and `building trip`. One reload, one test, diagnosis complete. Use bracketing traces rather than reading all possible code paths. [Cross-domain]

## Scars
- **The `existing_count < 2` limit** was placed in the code months ago (probably as a safety valve during early testing) and was never removed. It was invisible until vehicle 3 was tested. Any numeric limit in placement code is a candidate for accidental cap — search for `< N` patterns near placement calls before assuming "it should work for any count."

## Lessons for Allie

1. **`self.class` is `Module` inside a Ruby module's captured proc** — In a `mount_proc` block (or any proc defined in a module method), `self` is the module itself. `self.class` resolves to `Module`, not the module. All method calls via `self.class.xxx` silently fail (the methods don't exist on Module). Always use `self.xxx` or the module constant directly. Check any proc-based server or callback code for this pattern. [SketchUp] [Cross-domain]

2. **`__dir__` returns nil when loaded via SketchUp's `load` mechanism** — `__dir__` is nil; `File.expand_path('..', __FILE__)` is the correct pattern. All plugin files that construct paths should use `__FILE__`-based construction. This applies to any new file added to the plugin. [SketchUp]

3. **A successful `✅ JPods plugin loaded` message does not mean the file was re-read** — The success message comes from the module registration block guarded by `$jpods_registered`. If the load fails silently (or is never executed because of a line-break in the paste), the old in-memory module is unchanged. Always verify by checking a new `puts` line or a new constant defined in the file you're reloading. [SketchUp]

4. **WEBrick is not bundled with SketchUp's Ruby runtime** — Do not assume standard Ruby library availability in SketchUp. Before writing any standard-library code for the plugin, check `JPODS_WEBRICK_AVAILABLE`-style detection. UI::HtmlDialog with `add_action_callback` is the correct transport for HTML↔Ruby communication in SketchUp. [SketchUp]

5. **HtmlDialog callbacks always run on the main thread** — This is both the safety guarantee (model API calls are safe) and the risk (long operations block the UI). All four trip callbacks delegate immediately to DispatchServer and return quickly. Any operation that walks model entities should be pre-cached or deferred. [SketchUp]

## Scars

- **`self.class` in module proc** — Methods silently returned nil. No error, no warning. The rescue block swallowed the NoMethodError. The bug existed in every WEBrick endpoint for the entire session before diagnosis. Pattern: if an endpoint's response is always nil/empty and no error appears, check whether `self` in the proc has the class you expect.

- **Line-break reload command** — Ruby console parses Enter as execute. A multiline paste splits `load\n  Sketchup.find_support_file(...)` into two statements. `load` with no args → ArgumentError. The plugin appeared to reload (printed `✅`) because the constant was from a prior session. Never paste reload commands with newlines. Put the one-liner in CLAUDE.md and in each session handoff.
