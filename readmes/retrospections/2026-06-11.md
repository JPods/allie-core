### Session 3: CCW violation root cause and direct compaction

**CCW violation fix — `_platform_pts_entry_first` ps1 anchor:**

JPods_station_parking has no `gw_platform_parking` track. The fallback in `_platform_pts_entry_first`
used pod position to orient the platform pts. Once a pod advances past the midpoint, d_rev < d_fwd
and the orientation reverses. Every advance for ps6+ was dispatched in the wrong direction — toward
the entry end instead of the exit end.

The correct anchor: Sally's ps1 slot_position. Sally.init_from_model leaves the pts un-reversed
when `gw_platform_parking` is absent. ps1 was computed on that same direction. Using ps1 as the
orientation anchor in `_platform_pts_entry_first` guarantees the two functions agree on which
end is the entry — regardless of where the pod currently sits on the track.

**Direct-to-final-slot advance — `_advance_platform_queues`:**

Previous algorithm: find topmost empty slot with occupied slot directly behind it → advance one
slot per tick. With cap=9 and a pod at ps2, 7 ticks (7 seconds) to reach ps9.

New algorithm: sort occupied pods DESCENDING, assign compacted targets (rank 0 → cap, rank 1 → cap-1,
…), dispatch one per tick with 1s stagger. Highest-slot pod goes first — it clears the track.
Lowest pod follows 1 second later. O(n) ticks instead of O(n²).

With cap=9 and 2 parked pods: tick 1 → highest to ps9, tick 2 → remaining to ps8.

**Stale registry cleanup:**

`advance_pod_slot` now calls `Sally.release_slot` if the pod is missing from @@pods. Stops
repeated "not in active pods" log spam and unblocks the exit slot for subsequent pods.

**compact_platform_idle nil guard:** same fix as confirm_slots from session 2.

## Scars

- **Pod position is the wrong orientation anchor** — It works when the pod is at the entry end
  (just arrived). It fails as soon as the pod has advanced past the midpoint. The correct anchor
  is whatever reference Sally used to BUILD the slot positions — in this case, the raw pts direction
  locked in at `init_from_model`. ps1 slot_position IS that reference, available at runtime.

- **_advance_platform_queues scope** — This function runs for every station in the network on
  every 1s tick. Any bug here affects all stations simultaneously. The new algorithm is simpler
  than the spot-by-spot scan and handles all cases: single pod, multiple pods, pods already at
  target slots (target == from_slot → skip).

## Lessons for Allie

1. **Platform pts orientation must match Sally.init_from_model's frame** — When `gw_platform_parking`
   is absent, Sally uses raw pts direction. `_platform_pts_entry_first` must use the SAME direction
   or advance maneuvers travel backward. ps1 slot_position is the correct anchor — computed by Sally
   on that direction, available at runtime. Do not use pod_pos as orientation anchor for any pod
   that may have advanced past the midpoint. Domain: [SketchUp]

2. **Direct compaction: highest slot first** — When compacting pods toward the exit end, the pod
   closest to exit must depart first. If lower-slot pods depart first, they travel through upper-slot
   pods. Sort descending, advance one per 1s tick. The stagger at pod speed (~5-8 m/s) gives ~5-8m
   separation at the time each pod starts — well above the 3m personal space threshold. Domain: [SketchUp]

3. **Stale registry entries: fix at detection** — When @@pods and Sally's slot registry diverge,
   fix it where you first observe it. `advance_pod_slot` has both sid and nora_id — one line to
   call `release_slot`. Do not let the loop repeat the same stale-entry error every second. Domain: [SketchUp]
