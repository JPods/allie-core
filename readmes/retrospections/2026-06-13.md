# Retrospection — 2026-06-13

## What Was Done

Continued from June 12 session. Focused entirely on `platform_shuffle` test in
`jpod_console.rb`: placing three pods so the exit-slot pod (ps9) runs hold_loop
while the other two stay as occupancy probes.

Three separate root causes diagnosed and fixed:

**1. Entity lookup — wrong Ruby objects**
Custom `find_vehicles` console function recurses into `e.definition.entities`,
finding entities inside the component definition (shared template), not model
instances. `all_nora_vehicles_in_model` uses `model.entities` — different objects,
different entityIDs. Writing attributes to definition entities has no effect on
animation behavior which reads instance attributes.

**2. Test harness erase-recreate invalidates all manual patches**
`platform_shuffle` erases all `station_test=true` entities and recreates them fresh
on every animation start. Any attribute edits made after creation are discarded at
the next restart. The `tag_vehicle` lambda is the only place to set initial state.

**3. Exit-slot assumption baked in for 3-slot platform**
Original test placed "front" vehicle at slot_dists_in[2] = ps3, which is the exit
slot on a 3-slot platform but deep interior on a 9-slot. Sally's exit guard correctly
blocked dispatch. Fixed by using `pslots.last(3)` to get actual ps7/ps8/ps9.
Compact behavior confirmed working: Bill said "Excellent parking behavior."

**4. plat_pts exit-first for station_parking**
`gw_platform_parking` orientation fallback doesn't fire for station_parking templates.
plat_pts stays exit-first from geometry.json. dist_mm values are entry-first (measured
from entry). Walking 21250mm from exit end lands at ps1, not ps9.
Fixed by pre-initializing Sally, then using `slot_positions_for_station(sid)[1]`
(Sally's already-oriented ps1 world position) to detect and reverse plat_pts if needed.
Added `slot_positions_for_station` accessor to jpod_sally.rb.

## Lessons for Allie

1. **Component definition ≠ model instance** — `e.definition.entities` finds shared
   template objects. `model.entities` finds placed instances. These are different Ruby
   objects. Writing attributes to a definition entity silently writes to the shared
   template, not the specific pod in the scene. Always use `all_nora_vehicles_in_model`
   or `model.entities` search to modify placed instances. Confirm entityIDs match before
   trusting that a write took effect. [SketchUp]

2. **Test harnesses that erase-recreate invalidate all post-creation patches** — If
   a test erases and recreates entities on every start, the setup lambda (tag_vehicle)
   is the ONLY place to set correct initial attributes. There is no way to pre-patch
   entities that don't exist yet. Before manually editing entity attributes in console,
   ask: does this test recreate entities on start? If yes, fix the setup code instead.
   [SketchUp]

3. **Exit-slot guard is correct; test setup assumptions are not** — Sally's
   `_dispatch_next_sequential` exit-slot guard (`current_slot == cap_slot`) is the
   right design. When dispatch appears blocked forever, check test setup first: is the
   runner actually AT the capacity slot? Do not weaken the guard. Fix the placement.
   [SketchUp]

4. **plat_pts orientation is template-specific; never assume entry-first** — For
   station_parking (no gw_platform_parking), plat_pts loads exit-first from geometry.json.
   The existing orientation block is silently skipped. Any code that walks plat_pts to
   place vehicles or compute distances must verify orientation before use. Use Sally's
   `slot_positions_for_station(sid)[1]` (ps1 world position) as the orientation anchor —
   it is correctly computed by init_from_model Pass 3 regardless of geometry.json order.
   [SketchUp]

5. **Sally's slot positions are the authoritative world-coordinate anchor** — Sally's
   Pass 3 in `init_from_model` uses `gw_platform_in*` fallback chain to orient slot
   positions entry-first, then stores them as world Geom::Point3d in `@@slot_positions`.
   These positions are more reliable than plat_pts orientation because they use the
   actual connection geometry as anchor. Prefer `slot_positions_for_station(sid)[n]`
   over computing arc-distances from potentially mis-oriented plat_pts. [SketchUp]

## Scars

**Component definition entity write is silent and wrong** — Spent significant time
debugging why `all_nora_vehicles_in_model` returned different entityIDs than the
custom `find_vehicles` function. The symptom (attribute set, no error) was identical
to a successful write. No warning, no exception. The only tell was comparing entityIDs
between the two methods. Cost: ~2 hours of debug. Rule: always verify entityIDs match
between the lookup and the consumer before concluding an attribute edit worked.

**Erase-recreate harness makes console patches useless** — Spent time setting
attributes on correctly-found instances, confirmed they took effect, restarted
animation, attributes gone. The erase loop at test start was invisible until reading
the console code carefully. Cost: ~1 hour. Rule: read the test setup code before
patching entity attributes.
