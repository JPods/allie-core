# Retrospection — April 17, 2026

## JPods SketchUp Plugin — Bezier tangent direction bug resolved

### What was done
Resolved a persistent Bezier curve direction bug in the JPods SketchUp 2026 plugin
where some connection curves looped past their destination CP and approached from the
rear, violating the 15% grade curvature rules and producing routes far longer than
the direct path.

### Root cause
Two CP detection methods stored tangents with opposite conventions:
- `detect_gate_lines` (circle-based): stores **outbound** (radially away from structure origin)
- `pair_stubs` (track endpoint fallback): stores **inbound** (`ctrl[2]` = "into track")

Any global `.reverse` in the Bezier formula fixed one group and broke the other,
producing exactly the mixed results observed (S008.CP0, S012.CP1 etc. correct;
S001 and S008.CP1 wrong).

### Fix
Dot-product direction check at Bezier build time — snaps each tangent to the
geometrically correct direction regardless of stored convention:

```ruby
chord_v = p1 - p0
t0 = raw0.dot(chord_v) >= 0 ? raw0 : raw0.reverse  # must face toward p1
t1 = raw1.dot(chord_v) <= 0 ? raw1 : raw1.reverse  # must face toward p0
```

Applied in three places: `tangent_curve_pts` (build), `draw()` inline (preview),
`bezier_preview_pts` (hover). Also normalised `pair_stubs` to store outbound tangents
for consistency.

### Lesson
When two code paths produce the same data type but with opposite sign conventions,
a global sign fix is always wrong. The correct fix is a geometric invariant check
at the use site — the dot product with the chord vector is the right test here.
