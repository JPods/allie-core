## Lessons for Allie

1. **Deliberation before implementation is now proven** — Draft schema → three-stage deliberation → incorporate → implement. The sequence prevented 11 regressions before any code was written. Field renames caught in deliberation cost minutes; field renames caught in production cost days. Domain tag: [Cross-domain]

2. **followme_hash is the stale-map signal** — SHA-256 of raw followme.json file bytes. Map and trip files derived from a stale followme are invalid. TripPlanner checks: map mtime vs followme mtime, plus hash comparison. If hash mismatches, reject even if timestamps look OK. Domain tag: [SketchUp]

3. **ID notation: underscore (followme) vs dot (map v2)** — followme.json uses `S048_gw_platform`; map.json v2 uses `S048.gw_platform`. The regex `m = cid_s.match(/\A([A-Z]\d+)_(gw.+)\z/)` converts at write time. `build_map_lookup` registers both aliases at read time. Any new consumer of map.json must handle dot notation as primary. Domain tag: [SketchUp]

4. **UTC mandate: 9 local-time fields found across 6 files in one session** — The audit pattern works: grep for `Time.now.strftime` without `.utc`, then verify it's a data field not a display string. Ruby `Time.now.strftime` without `.utc` is the canonical bug signature. Domain tag: [Cross-domain]

5. **v2 segment records are Hashes, not Strings — consumers must guard both** — Any code that iterates trip.json segments must handle both `seg.is_a?(Hash)` (v2) and plain String (v1). This transition period will last until all consumers are upgraded. The animator fix (`seg['id'].to_s` vs `seg.to_s`) is the template for every other consumer. Domain tag: [SketchUp]

# Retrospection — 2026-05-20 (Session 2)

## What Was Done

### Approach Curve False Violations — Fixed

Noelle was blocking builds with 5 false BLOCK violations at ~1.2m approach curve radius on S048, S050, S051. All visually correct (3.5m+). Three-attempt arc:

**Attempt 1:** Added 0.3m chord filter. Violations moved from 0.9m to 1.2m — still BLOCK.

**Attempt 2:** Added `APPROACH_SKIP_DEPTH = 2.0m` constant (added to `jpod_constants.rb`) and skip zone in `check_approach_curves`. Still 1.2m after full SketchUp restart.

**Root cause:** `check_approach_curves` was calling `vehicle_path_for`, which calls `stitch_structure_followme_paths` — prepends/appends station internal guideway paths (U-turns, platform approaches) to the inter-station beam path. The U-turn geometry, which has genuine tight radius, was polluting the measurement.

**Fix:** Changed to `base_vehicle_path_for` — reads only the guideway's own `beam_path` attribute, no structure stitching. Result: "all guideways approved (min radius >= 8.0 m)."

TFTS: `process/inbox/20260520T232624-tfts.md`

### Waypoint (via_markers) Dropping — Fixed

The 3 waypoints placed between S048 and S050 were being dropped during Build. Topology showed `[1,2]` instead of `[1,2,3]`. Guideways built straight; columns appeared at waypoint positions as a separate pass.

**Root cause:** `generate_feature_json` was creating TWO `cp_` entries per bidirectional connection — one for each direction. `build_from_config` called `build_segment` twice for the same physical guideway pair. `build_segment` with `replace:true` erases both `seg_fwd_id` and `seg_rev_id` before rebuilding. The second call always won, stamping the second entry's `via_markers: []` and destroying the first call's work.

**Fix:** `generate_feature_json` now checks if the reverse `cp_` key already exists before creating a new entry. Both `seg_` entries nest under ONE `cp_` key per physical pair. `build_from_config` calls `build_segment` once. feature.json: 6 cp_ entries → 3. Trip S048→S050: 465m → 556m.

Added warning to TFTS: future single-guideway (one-way) connections require revisiting the merge logic before implementing — document the case first.

TFTS: `process/inbox/20260520T234607-tfts.md`

## Lessons for Allie

1. **`vehicle_path_for` is for animation; `base_vehicle_path_for` is for geometry** — Any check that measures a specific guideway's own spatial properties must use `base_vehicle_path_for`. `vehicle_path_for` stitches neighboring station geometry and will produce false readings for any per-guideway spatial analysis. Domain tag: [SketchUp]

2. **Ask "what geometry is this measuring?" before tuning the algorithm** — Two fix attempts (chord filter, skip depth) were wasted tuning circumradius measurement parameters when the geometry source was wrong. The correct question at the first anomaly: "Is the path I'm measuring the right path?" Domain tag: [Cross-domain] — applies to any sensor, metric, or spatial check.

3. **ONE cp_ entry per physical guideway pair — duplicate entries cause double-build** — `build_segment` builds BOTH tracks (forward and reverse) in a single call. A second `cp_` entry for the reverse direction causes a second `build_segment` call on the same pair. The second call always wins, erasing `via_markers` from the first. `generate_feature_json` must check for the reverse key before creating a new entry. Domain tag: [SketchUp]

4. **`build_segment` with `replace:true` is total erasure for the pair** — It erases ALL guideways whose `connection_id` is in `[seg_fwd_id, seg_rev_id]` — both directions. This is by design (prevent orphaned geometry). Any code that calls `build_segment` must only call it ONCE per physical pair or it will overwrite itself. Domain tag: [SketchUp]
