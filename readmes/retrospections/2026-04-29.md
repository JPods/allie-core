2. Recompute CPs.
3. Re-export FollowMe.
4. Confirm strict export diagnostics show non-zero platform detections for each station.
5. Run Noelle validation before Natalie routing.

## SketchUp plugin regression — CP datum direction + missing FollowMe bridge (late session)

### What changed

- Guideway alignment at structure gates was corrected so joins are bottom-centerline to bottom-centerline.
- FollowMe structure collector was expanded so station bridge segments at `stub_pair` gates are no longer silently skipped.

### Root causes

1. Paired `stub_pair` midpoint correction used the right magnitude (`BEAM_WIDTH / 2`) but direction could resolve outward.
2. FollowMe collection logic was overly strict:
	- exact structure-name match (`JPods Structure`) only
	- track-entity detection by name containing `track` only
	- valid `stub_pair` bridge entities were ignored.

### Fixes implemented

- Enforced inward `BEAM_WIDTH / 2` correction toward structure origin/interior for paired `stub_pair` CP midpoint.
- Structure collectors now use JPods structure-candidate detection (attribute/tag based), not just literal instance name.
- FollowMe track detection now accepts entities tagged `stub_pair` (in addition to existing track-name paths).

### Lesson

Datum correctness is directional, not only scalar. A correct `0.25 m` shift in the wrong direction is still wrong.
Also, naming-only discovery causes silent topology loss; FollowMe path extraction must honor tags/attributes as authoritative.

## SketchUp plugin regression — CP0/CP1 asymmetry (final repair, same day)

### What changed

Bill observed: "all the stations CP0 end have the CP offset. We are not calculating the CP the same for both ends."

Two distinct bugs were present in `detect_cps_from_stub_pair_tags`:

### Bug 1 — `out_dir = cross_v.cross(Z)` (scan-order-dependent sign)

`cross_v` = tip_b − tip_a. The sign of `cross_v × Z` depends on which tip the entity iterator gives first. For one gate it pointed outward; for the other gate (opposite scan order) it pointed inward. This caused CP0 to have a rotated tangent at every station, consistently — same gate index, same fault, every time.

**Fix:** Replaced with the normalized average of the two stubs' own tangents:
```ruby
avg_tan = Geom::Vector3d.new(
  (ta[:tangent].x + tb[:tangent].x) / 2.0,
  (ta[:tangent].y + tb[:tangent].y) / 2.0, 0
)
out_dir = avg_tan.normalize
```
Each stub tangent = `(outer_pt − inner_pt).normalize`, which is always outward for that stub, regardless of scan order.

### Bug 2 — BEAM_WIDTH/2 cross-track shift applied after midpoint

`gate_ctr = midpoint(outer_pt_a, outer_pt_b)` is already the correct CP position — the centroid of the gate-end face cluster IS the beam CL, so the midpoint IS the pair CL. An additional `BEAM_WIDTH/2` shift in the cross-track direction moved the CP 0.25 m off-center. At CP1 (which had a correct tangent direction), this produced a consistent 0.25 m gap or overlap at the seam.

**Fix:** Cross-track shift removed entirely. No correction applied after midpoint.

### Regression guard written

Full regression guard document created at:  
`/Users/williamjames/Allie/readmes/sketchup/jpods-cp-regression-guard.md`

Covers: the invariant (midpoint is authoritative), all four failure modes, Ruby Console diagnostic checks, and a corrective-action table by symptom.

### Lesson

When a paired geometry midpoint gives the correct position by construction, **no correction should follow it**. Any offset applied afterward is a regression by definition. The midpoint of two stub beam-CL gate-end centroids is already the pair CL at the seam.

The scan-order dependency of `cross_v × Z` is a recurring trap. Any vector formed as `A − B` where A and B are determined by iteration order will have an unpredictable sign. Always use geometry-intrinsic direction (stub tangent from inner to outer) instead of cross products of paired-point differences.
