### Coverage (Walk-Ride-Walk) fixes

**Problem 1 — Odd islands of capacity:**
Small destination-station circles appearing as isolated blobs, unconnected from the
main coverage area. These were real results (reachable destinations) but too small to
be meaningful and visually indistinguishable from simulation artefacts.

**Fix:** `MIN_RADIUS_M = 300` in `timemap.js` — any destination circle with less than
300m of remaining walk-from-station budget is suppressed. Legitimate large coverage
areas are unaffected.

**Problem 2 — Phantom ride times:**
Potential for near-zero ride times from simulation edge cases (disconnected-island
stations with spurious entries) to produce huge coverage blobs at distant stations.

**Fix:** Speed filter in `_draw()` — any `trip_stats` entry where implied pod speed
exceeds 200 km/h (3333 m/min) is rejected. Confirmed: simulation already logs a
warning and skips the trip on `find_path()` returning None, so disconnected stations
produce no `trip_stats` entries. Filter is a safety net.

**Problem 3 — Nearest station invisible when beyond budget:**
When the clicked point was more than 30 min walk from any station, no marker appeared
at the nearest station. User saw only walk circles at the click point with no context.

**Fix:** `_boardingMarker` always drawn at the nearest station:
- Orange dot when reachable within budget
- Red dot when beyond 30-min budget, tooltip: "⚠ beyond 30 min budget"
- Status bar shows: `⚠ Nearest station ST_X is 34.2 min walk — beyond 30 min budget`

**Problem 4 — Coverage clicks blocked by map layers:**
Clicking on a guideway line or internal structure line while in Coverage mode fired
`L.DomEvent.stopPropagation(e)` before TimeMap received the click. Coverage only
worked on empty map background.

**Fix:** Check `TimeMap.isActive()` at the top of every line and CP click handler.
If active, delegate to `TimeMap.handleClick()` and return. Now coverage works by
clicking anywhere on the map — empty space, guideways, structure lines, or CP dots.

**Files changed:**
- `route_time/gui/static/timemap.js` — MIN_RADIUS_M, phantom filter, boarding marker, status
- `route_time/gui/static/app.js` — TimeMap delegation in line + CP click handlers

### Confirmed bug — Adjacent station travel time inversion

**Observed behavior:**
Running the simulation on a grid network and checking the results table or the
Walk-Ride-Walk coverage circles shows:

- An **adjacent station** (one block away) reports travel time **> 30 minutes**
- **Stations further away** (multiple blocks) report travel times **< 30 minutes**

This is physically impossible in a well-connected network. Under normal conditions,
a closer station must have a shorter or equal travel time. The only valid exceptions
are documented below.

**What should be true (design invariant):**
> In a non-congested network, travel time to station B should be monotonically
> non-decreasing with geographic distance from origin A, along the dominant
> direction of travel. A nearby station cannot require a longer ride than a
> distant one in the same direction unless routing is forced around the long way.

**Likely root cause — one-way routing over-length:**
JPods traffic circles are counter-clockwise one-way. In a grid, to travel from
station A to an immediately adjacent station B that is in the "wrong" direction
relative to circle rotation, Dijkstra must route the pod through a very long path
(potentially all the way around the grid) rather than a direct short segment.

Example: if A is south of circle C1 and B is west of C1, the CCW routing may
require going north → east → south → west around the full circle before exiting
toward B — or, if C1's arms don't connect correctly, routing out of C1 and back
around multiple circles.

**Alternative cause — CP connection direction error:**
If the grid generator connected a CP arm in the wrong direction (e.g., circle east
arm connected to the east arm of the next circle instead of to the station's
appropriate CP), guideways would be reversed. Pods routing to the adjacent station
would be sent "backwards" through the one-way system, requiring a full grid traversal.

**Alternative cause — Station CP heading mismatch:**
If `_cp_by_heading()` selected the wrong arm for a circle CP (e.g., assigned the
north arm connection to the south arm due to a 180° ambiguity in heading matching),
half the connections in the grid would be backwards.

**Valid exception (not a bug):**
Massive congestion on a direct route can increase travel time. A distant station
on an uncongested alternate path may genuinely be faster. This is the correct and
desired behavior for capacity planning — the coverage circles SHOULD show smaller
circles for congested destinations so the designer can see where to add capacity.

**How to distinguish bug from congestion:**
- If the network was just generated (no traffic yet, first simulation run at low
  demand), congestion should be near zero. If adjacent station still shows > 30 min,
  it is a routing/topology bug, not congestion.
- Check `line_stats.congestion` for the guideways between A and B. If congestion ≈ 0
  and travel time is still > 30 min, the path Dijkstra found is wrong.
- Compare `route_line_ids` for the A→B trip vs the A→(far station) trip. The A→B
  route_line_ids count should be small (a few lines). If it is hundreds of lines, the
  routing is going the long way around.

**Next steps to diagnose:**

1. Generate a small grid (e.g., 3×3, 1-mile spacing) and run simulation
2. Print `trip_stats` for adjacent station pairs — check `median_trip_ms` and
   `route_line_ids` length
3. Inspect `route_line_ids` for an anomalous pair — trace which circles and stations
   the route passes through
4. Check `_cp_by_heading()` in `api.py` for a potential 180° assignment error:
   ```python
   # Current: minimum angular difference, wrapping at 180°
   # Risk: if two arms are exactly 180° apart, either could be selected
   ```
5. Verify grid connection logic in `POST /api/network/grid`:
   - N-S block: circle south arm → station CP_N; station CP_S → next circle north arm
   - E-W block: circle east arm → station CP_S; station CP_N → next circle west arm
   - Check: are east/west arm headings (90°/270°) correctly matched to station heading 90°?

**Files to inspect:**
- `route_time/gui/api.py` — `_cp_by_heading()` and `POST /api/network/grid` connection logic
- `route_time/engine/routing.py` — `find_path()` Dijkstra — verify it respects line direction
- `route_time/engine/simulation.py` — `trip_stats` `route_line_ids` for anomalous pair

### Design principle confirmed

**"Closer station should always have quicker access than remote ones unless there is
massive congestion. If there is congestion, we need to see it so networks can be
adjusted to have more capacity."**

The Walk-Ride-Walk coverage display correctly implements this:
- Congested path → higher `median_trip_ms` → less remaining walk budget → smaller circle
- The coverage circles are the visual signal for where to add capacity
- Adjacent station with a smaller coverage circle than a distant station = congestion
  or topology bug on that segment — actionable for network designers

This principle is a **design invariant** for the simulation and coverage display.
It should be verified after every grid generation or network edit.

### Diagnostic result — adjacent station inversion resolved

**`diag_grid.py` was run** on a fresh 3×3 grid (1-mile blocks). All 12 adjacent
station pairs (6 N-S + 6 E-W, both directions) route correctly:

- Direct direction: 13 lines, ratio 1.0× (straight through)
- Return direction: 19 lines, ratio 1.1× (normal turnabout detour)

**Conclusion: the grid generator topology is correct.** No connection direction bug.
No `_cp_by_heading` ambiguity in practice.

**Root cause of GUI inversion:** Congestion under load. When the direct A→B guideway
gets jammed, Dijkstra reroutes pods the long way, producing high travel time for that
O-D pair while uncongested longer routes stay fast. This is **correct behavior** — the
coverage circles accurately signal congestion. The design principle holds.

The bug documented in the previous section is **not a topology defect**.
It is the simulation correctly reporting congested segments. Network designers should
respond by adding parallel capacity on the congested guideway, not by questioning the
routing algorithm.
