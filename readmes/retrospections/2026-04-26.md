### Walk-Ride-Walk coverage circles (added same session — later context window)

**Feature:** Click any map point → colored coverage circles for 5/10/20/30 min total journey time.
- Green = 5 min, Blue = 10 min, Yellow = 20 min, Red = 30 min
- Algorithm matches Java `TimeGraph.java`: find closest boarding station, walk there, ride to each dest station, walk remaining budget from dest
- `remaining = budget − walk_to_boarding − ride_time(boarding→dest); radius = remaining × walk_speed_m/min`
- Uses `turf.circle()` (64-step polygon) per destination, `turf.union()` to merge all same-color circles — only outer boundary renders, no interior lines
- Transparency (opacity) user-configurable in Settings → Coverage circles (default 0.15)
- Legend appears bottom-left while mode active; Esc exits

**Files changed:**

### Third session — batch .jpd conversion (same day, continued context)

**Objective:** Convert 520 legacy Java Route-Time .jpd files to new GUI format (with StructureMeta).

**Root problem:** Old files used sequential node naming (`SW1xxx`, `ST1xx`) with a `<Groups>`
block (station templates as `<tmpl>`, circles as `<Circle C1-C4 D1-D4>`).
New GUI requires Structure/CP objects with proper inbound/outbound stub nodes.
The naming conventions are incompatible — `_reconstruct_structures_from_net()` only works
for files the new GUI created.

**Solution — `convert_jpd.py`:**
```
/Users/williamjames/Documents/08_JPods/03_Technology/route_time/convert_jpd.py
```
Run:
```bash
cd /Users/williamjames/Documents/08_JPods/03_Technology
python3 -m route_time.convert_jpd <input.jpd> [output.jpd]
python3 -m route_time.convert_jpd --all /Applications/RouteTime_JPods --dest /Applications/RouteTime_JPods/converted
```

**Algorithm:**
1. Parse `<Groups>` to identify station templates (tmpl) and circles (Circle)
2. For each circle: compute centroid of C1-C4+D1-D4 nodes; infer 4 arm headings by
   bearing from centroid to each Cn node (snapped to 45°); call `build_traffic_circle()`
3. For each station: use ST node position as center; split SW nodes north/south of center,
   compute heading from south-centroid→north-centroid bearing; call `build_station()`
4. Build fresh Network with proper internal topology (old nodes/lines discarded)
5. Detect inter-structure connections: find old lines crossing structure boundaries;
   match each to nearest unconnected CP pair by geographic distance
6. Call `connect_cps()` for each matched pair; save with `save_jpd()` + StructureMeta

**Result:**
- 397 files converted successfully (0 errors)
- 28 "files" were empty directories (Java saves that crashed/never wrote) — skipped
- Output at: `/Applications/RouteTime_JPods/converted/`
- Spot-check: Kenya (38 stations, 19 circles, 152 CPs, 60 pairs connected) — clean round-trip
- Small networks (single-station sketches) convert correctly with 0 pairs connected

**Quality metrics per converted file:**
- All structures have correct type (station | traffic_circle)
- All CPs have inbound_node, outbound_node, heading_deg, center_lat/lon
- connected_to is symmetric (A→B iff B→A)
- Open CPs on outer network edges are expected (not defects)
- Short CP-pair distances (< 20 m) indicate adjacent structures — plausible, check visually
- Very long connections (> 3000 m) may indicate a mis-match — flag for review

**Known limitations:**
- All circles infer arm headings from geometry — arms that don't align to 45° will be
  snapped; for networks with diagonal arms (45° rotation), this works correctly
- Heading inference for east-west stations uses northernmost SW node: correct for N-S grids;
  may need manual correction for truly east-west oriented stations
- Multiple connections between the same pair of structures (parallel guideways) are
  handled by the `n_connections = max(1, line_count // 2)` heuristic — verify in complex
  interchange areas
