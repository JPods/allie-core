Path attribute write — four attempts:
1. Array of Geom::Point3d → SketchUp cannot serialize; rescue nil silently swallowed
2. Nested [[x,y,z],...] arrays → SketchUp serializes to String; Priority 1 Array check fails
3. Flat Float array [x,y,z,...] + direct get_attribute('JPods','jpods_path') → works for gw_lift_in

Root cause of gw_platform_in failure: SketchUp Group entity identity.
`_index_gw_entities` traverses `model.entities` (indexer path).
`_scan_gw_entities_recursive` traverses `inst.definition.entities` (scanner path).
For this specific Group, they resolve to different Ruby proxy objects. Writing to one does not
set the attribute on the other. This is a SketchUp internal entity model constraint.

**Resolution: SCAN classification**
FollowMe solid tracks (>20 declared pts, <1/3 found by scanner) → SCAN status, not SEVERE.
Parallel to ARC classification for arc guideways. extracted.json is authoritative.

## Lessons for Allie

1. **SketchUp set_attribute type constraints** — Only primitive Ruby types survive round-trip:
   Integer, Float, String, Bool, flat Array of those. Geom::Point3d → silent failure.
   Nested Ruby arrays → stored as String. Flat Float array is the only safe serialization.
   Domain: [SketchUp]

2. **Group entity identity divergence** — SketchUp Group objects accessed via `model.entities`
   traversal and `inst.definition.entities` traversal are different Ruby proxy objects for the
   same logical entity. Writing an attribute to one does not make it visible on the other.
   This is specific to Groups (unique definitions); ComponentInstances sharing definitions
   don't have this problem. Test: if scanning finds 0 for an attribute just written, suspect
   Group entity identity. Domain: [SketchUp]

3. **SCAN vs SEVERE — accept scanner limitations** — When a scanner heuristic cannot verify
   complex FollowMe geometry, the right response is classification (SCAN) not escalation (SEVERE).
   The declared data (extracted.json) is the authority; the scanner is advisory.
   Over-engineering the scanner creates fragile coupling to SketchUp internals.
   Domain: [SketchUp]

4. **Rescue nil is dangerous in diagnostic paths** — `set_attribute ... rescue nil` silently
   swallows failures. Always use explicit rescue with a print when diagnosing write paths.
   The silent nil was the reason two fix attempts were wasted before the root cause was found.
   Domain: [SketchUp] [Cross-domain]

## Scars
