## Lessons for Allie

1. **`gw_far_out` is a departure track segment, not the platform** — `gw_far_out` appears only in `1cp_line_end` departure paths (`gw_platform → gw_uturn_1 → gw_far_main → gw_far_out → gw_stub_pair_0_out`). It is never the platform destination. `1cp_simple` stations reach their platform via the same tags as all other station types: `gw_platform_in → gw_platform_parking → gw_platform`. Domain: [SketchUp] [Route-Time]

2. **The distinction between `1cp_line_end` and `1cp_simple`** — Both are end-of-line single-CP stations. The difference is in the departure path: `1cp_line_end` needs a reversal loop (`uturn_1 + far_main + far_out`) to face outbound again. `1cp_simple` exits directly via `platform_out → stub_pair_0_out`. If you see `gw_uturn_1` and `gw_far_main` tags, it is `1cp_line_end`. If you see only stub/platform tags, it is `1cp_simple`. Domain: [SketchUp]

3. **Two sources of truth for the same computation is a design defect, not a bug** — `build_platform_round_trip` and `TripPlanner` were computing trips independently. Bill named this directly: "There should only be one source of truth." The fix was not to patch the outputs to match, but to make one call the other. When two code paths produce the same output from the same input, one must be eliminated. Domain: [Cross-domain]

4. **Bill will review trip.json segment-by-segment and mark errors with "wrong"** — Format: he prefixes the bad record with `"wrong"` then provides the correct record. Read these as axiomatic corrections — the routing tables in `TripPlanner` must match them exactly. Domain: [SketchUp]

## Scars

- **`gw_far_out` as platform destination** — coded this as the platform for `1cp_simple` based on the comment "far_out contains the platform." That was wrong. `gw_far_out` is a track section in the departure loop — not named for containing the platform. Cost: two regenerate cycles.

- **Connection IDs synthesized as bidirectional** — `adjacency` created reverse edges by reusing the forward connection ID. followme.json has separate entries for each direction (`seg_S048_cp1_S050_cp0` and `seg_S050_cp0_S048_cp1`). Synthesizing the reverse edge with the forward ID put the wrong guideway in the trip. The fix is one line: remove the reverse-edge push in `adjacency`. Cost: one regenerate cycle.

- **`1cp_simple` departure** — first fix set it to `platform_out → stub_pair_0_out` (no u-turn), based on a wrong assumption that S050 was simpler than S051. The physical network shows both need the same departure loop. Cost: one more regenerate cycle.

## Lessons for Allie (Part 2)

5. **`track_idx=1` = from→to travel direction** — Confirmed in Build pipeline code comment. When `prev_last` is nil in `resolve_gw_segs`, always prefer `track_idx=1` candidate. Picking `candidates.first` causes CW motion because track_idx=0 has reversed points. Domain: [SketchUp]

6. **gw-only is a hard constraint, not a preference** — Every attempt to bridge Build gaps through structure tracks produced wrong segment insertions. The gap is real and Noelle must fix it in the Build pipeline. Animation code must log and skip, never bridge. Domain: [SketchUp]

7. **Segment name encodes direction** — `seg_S048_cp1_S050_cp0`: first station = from, second station = to. This is parseable. Combined with `track_idx=1`, it gives two independent direction signals. If they conflict, the Build pipeline has a bug. Domain: [SketchUp]
