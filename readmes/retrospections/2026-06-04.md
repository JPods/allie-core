## What Happened

### station_thru_dip

Straightforward extraction cycle. Six severe junction gaps on first run. Most were endpoint snaps in SketchUp — the model geometry was close but not connected. The 380mm gap on gw_far_main→gw_cp_out_lead_0 was different: the track was geometrically too short and needed to be extended, not just snapped. Three passes → clean.

### traffic_circle7

More complex. Four separate problem layers, each blocking the next:

**Layer 1 — code didn't know about pass_chains.** The `_validate_connectivity` function only had logic for platform-station topology (hold_loop + landing_chains + exit_chains). traffic_circle7 uses `pass_chains` instead — a completely different routing model for a through-traffic circle. The function hit no chains and returned without checking connectivity. Added a `has_pass_chains` branch that reads pass_chains instead.

**Layer 2 — `landing_chains.note` treated as a chain.** The `"note"` metadata key inside the `landing_chains` dict was being iterated as if it were a CP entry. Produced "unrecognised format (String)" warnings. Fixed with a one-line `next if cp == 'note'` guard.

**Layer 3 — CP numbering mismatch.** The gw_cp_in_N tags in SketchUp were at wrong CP positions. gw_cp_in_0 was at what cp.json called CP3, gw_cp_in_2 was at CP0, etc. — a systematic rotational offset in the SketchUp model. This produced junction gaps of exactly 11,042mm and 17,254mm (geometrically predictable distances, not random noise). Diagnosis: the gap *sizes* identify the cause. Fix: retag all 16 components in Entity Info.

**Layer 4 — connector track naming convention.** Bill renamed `gw_N_in`/`gw_N_out` → `gw_in_N`/`gw_out_N` for consistency with the `_#` suffix convention. Required updating all references throughout lines.json. A text-replacement artifact left a stray double-quote that broke JSON parsing.

After all four layers: ✓ 112 ok, 0 warn, 0 severe.

## Lessons for Allie

1. **Junction gap size reveals cause, not just severity** — Gaps of 1–40mm are alignment errors (missed snaps). Gaps of 11,042mm or 17,254mm are tag mismatches: the geometry is at the right position but the tag says it's somewhere else. When you see large, exact gaps on a traffic-circle or rotational template, look at CP numbering in Entity Info, not at the geometry endpoints. [SketchUp]

2. **pass_chains is a topology, not just a field** — traffic_circle7 has no platform, no hold loop, no landing/exit chains. It uses `pass_chains` (from_cpN_to_cpM) to describe all valid routes through the circle. Any code that validates or routes through templates must know about both topologies. If a template has `pass_chains`, it does not have `hold_loop_chain`, and vice versa. [SketchUp]

3. **Metadata keys in chain dicts need explicit skips** — `landing_chains` and `pass_chains` both have a `"note"` key at the top level for human-readable documentation. Any iteration over these dicts must skip `"note"` (and potentially `"circle"` in pass_chains). The pattern `next if key == 'note'` should be applied wherever these dicts are iterated. [SketchUp]

4. **Axis unification is template-only** — The Ruby script that bakes instance rotation into definition geometry preserves relative positions but shifts absolute world coordinates. This is safe for templates (absolute world position doesn't matter). It would be catastrophic in a network model where all station instance positions would shift. Never run this script against a network model. [SketchUp]

5. **Designer risk list format** — `readmes/sketchup/jpods-template-designer-risks.md` is the new canonical reference for template authors. When Extract Template produces a warning or error that isn't immediately obvious, look here first. The 9 entries cover the real defects encountered in the 2026-06-04 extraction session. [SketchUp]
