## Why These Items

**C-W24-1 (other templates clear on first try):** The vi_entity scan and `_vi_component_direction`
are template-agnostic. Once Bill adds the vector_in component to each gw_cp_in instance in the
other templates, the code finds it via proximity. No per-template adjustment needed.

**C-W24-2 (normalize_text tuning):** The brief generator strips per-run variable data, but the
flag messages include the specific formation name and CP index. Revision numbers in error context
strings (from the `(rev N)` extraction log) would not be in flags, but the CP-specific text
("CP0: gw_cp_in_0 missing...") normalizes correctly. The real risk is that 16 revisions of
RED FLAG for the same root cause appear as 16 separate observations with slightly different wording.

**C-W24-3 (trial5 cleanup):** Any template folder without lines.json will cause the batch
Extract Template run to emit a `no lines.json` warning and fail silently. That's not harmful but
is noise. trial5 is almost certainly from a prototyping session.

**C-W24-4 (5m proximity too loose):** The threshold was set for JPods_station_parking where the
two CPs are ~74m apart. For a closer arrangement, a 5m bubble could overlap two different CPs.
The correct fix: nearest-wins (sort by distance, assign the closest vi_entity to each cp_in,
no vi_entity used twice).

**C-W24-5 (arc Z span demand):** The `gw_uturn_*` Z-span check in Noelle fires whenever the arc
has a slight Z variation (flange artifact). This has been noted as cosmetic in prior sessions but
has not been fixed at the model level. After 6 templates × multiple revisions it will be the
highest-frequency observation in the JSONL.
